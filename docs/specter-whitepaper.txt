SPECTER

Ignotus Nemo

Version 1.0.6
26 March, 2026

/**** Introduction ****/

I call my proposal Specter because, like a ghost, authorization in this
system exists but cannot be seen. There are no signatures. No public
keys used for signing. Only proofs of ownership-bound authorization
that leave no trace.

Bitcoin requires every node to download and verify every signature ever
made. This is already painful. With post-quantum cryptography, signatures
become 30-100x larger. The chain bloats to absurdity. Worse, every
signature reveals a public key, which reveals identity linkage.

Specter takes a different path. I do not compress signatures. I
eliminate them entirely.

The goals, in strict priority order:

  1. Minimum possible permanent chain data
  2. Non-interactive receiving (sender pays without recipient being online)
  3. Full post-quantum security (no elliptic curves, no pairings)
  4. Signatureless authorization (via zero-knowledge proofs)

This document is a research whitepaper and design specification. It
describes Specter's cryptographic architecture, security model, and
engineering targets. It is not a complete byte-level protocol
specification. Serialization formats, wire protocol details, and
finalized constraint layouts are deferred to the implementation
specification. Performance numbers are engineering targets unless
explicitly marked as benchmarked.

/**** The Core Idea ****/

Specter is a proof-authorized state machine. There are no transactions
at the protocol layer. There is only state: a set of unspent notes and
a set of consumed nullifiers. State transitions are authorized by
zero-knowledge proofs. Blocks are proofs that state evolved correctly.

Instead of proving "this signature is valid for this key," the system
proves:

  "The global state root evolved from A to B by a set of valid
   state transitions, each authorized by knowledge of a spending
   secret consistent with an on-chain commitment, with replay
   protection."

This is a semantic shift. Authorization becomes a zero-knowledge
statement about ownership. Signatures, public keys, and transaction
objects disappear from the protocol entirely.

The insight comes from observing what signatures actually do:

  1. Prove knowledge of a secret key
  2. Bind that proof to a specific message
  3. Prevent replay (via nonces or UTXO consumption)

All three can be achieved with zero-knowledge proofs over hash
functions. No elliptic curve discrete log. No lattice hardness for
authorization. Just hashes.

/**** Post-Quantum Security Model ****/

Specter's post-quantum security rests on three primitives:

  1. BLAKE3 -- for all non-ZK hashing (block headers, key derivation,
     non-circuit Merkle trees). 256-bit output provides 128-bit
     post-quantum preimage resistance under Grover's algorithm.

  2. Poseidon2 over BabyBear -- for in-circuit hashing only. The
     arithmetic-friendly construction required inside STARK proofs.
     Production parameters (state width 16, rate 8, capacity 8,
     8 external rounds, 13 internal rounds, S-box x^7) provide
     128-bit classical security. Post-quantum security of the
     in-circuit hash is approximately 64 bits, sufficient for ZK
     soundness (which requires binding, not preimage resistance,
     at the 64-bit level for our security parameter). Full-round
     Poseidon2 over BabyBear remains unbroken as of March 2026
     despite active cryptanalysis (ePrint 2025/937, 2025/950,
     2026/150 found attacks only on round-reduced instances).

  3. ML-KEM-768 (FIPS 203, August 2024) -- for note discovery only.
     Module-LWE based key encapsulation, providing 128-bit post-
     quantum security. This is the one non-hash primitive in Specter
     and is the minimum necessary for non-interactive receiving
     without sacrificing post-quantum security.

A note on the hash-only ideal: pure hash-only post-quantum note
discovery does not exist. No construction is known that allows a
recipient to identify outputs addressed to them without either (a) an
asymmetric primitive or (b) mandatory out-of-band communication for
every payment. The latter destroys non-interactive receiving. The
former requires choosing between elliptic curves (broken by Shor's
algorithm) and lattice-based KEMs (ML-KEM, standardized by NIST).
We choose ML-KEM. Every other component of Specter is hash-based.
Authorization proofs, nullifier set, UTXO accumulator, block proof,
and consensus are all pure hash primitives.

Throughout this paper:

  H(...)      denotes BLAKE3 (non-ZK contexts)
  H_zk(...)   denotes Poseidon2 over BabyBear (in-circuit use)

Both use domain-separated invocations via fixed-length context strings.

/**** Ownership Without Keys ****/

Each user holds a Root Entropy Value (REV). This is their master secret.
From it, everything is derived:

  Master Secret (REV, 256 bits):
    |
    +-- id_root    = H(REV || "specter.id-root")
    +-- spend_root = H(REV || "specter.spend-root")
    +-- null_root  = H(REV || "specter.nullifier-root")
    +-- auth_root  = H(REV || "specter.auth-root")
    +-- scan_key   = H(REV || "specter.scan-key")     [ML-KEM use only]

  Each root serves exactly one protocol role:
    id_root:    Derives coinbase ownership commitments
    spend_root: Derives owner_secrets for spending notes
    null_root:  Derives nullifiers for spend prevention
    auth_root:  Derives per-transition authorization bindings
    scan_key:   Seeds ML-KEM key derivation for note discovery

No single derived root can reconstruct any other, preventing
cross-protocol leakage. The REV never leaves the user's device.

All keys are deterministically derived from the REV alone. A user
who retains their REV (or its mnemonic encoding) can fully reconstruct
all derived roots and recover all unspent notes. This is the wallet
recovery guarantee.

When spending a note, the user proves in zero knowledge:

  C1. "I know owner_secret such that owner_commitment matches"
  C2. "I have bound this proof to this specific state transition"
  C3. "My nullifier is correctly derived in two stages from null_root
       and the note, and is fresh (not used before)"
  C4. "All domain tags are consistent"

Full state transition validity additionally requires:

  V1. Input existence (each input note_cm exists in the UTXO
      accumulator at the claimed pre_state_root)
  V2. Value conservation (sum of input values = sum of output
      values + fee)
  V3. Range validity (all values non-negative, bounded to 64 bits)
  V4. State-root consistency (all inputs proven against the same
      UTXO root; all nullifier non-membership proofs against the
      same nullifier root)
  V5. Canonical encoding (note serialization uniqueness)

/**** Note Model ****/

A note (UTXO) in Specter is a tuple:

  note = (owner_commitment, value_commitment, rho)

Where:
  owner_commitment = H(owner_secret || "specter.owner")      (32 bytes)
  value_commitment = H_zk(v || r || "specter.value")         (32 bytes)
  rho              = per-note randomness (256-bit, unique)   (32 bytes)

The note commitment (canonical identifier in the UTXO accumulator) is:

  note_cm = H(owner_commitment || value_commitment || rho || "specter.note")

This single 32-byte value enters the MMR accumulator. The nullifier
binds to it via two-stage derivation:

  sk_note   = H(null_root || rho || "specter.note-null")
  nullifier = H(sk_note || note_cm || "specter.nullifier")

Two-stage derivation provides: (1) nullifiers unique per note via rho,
(2) the nullifier commits to the full note commitment preventing cross-
note confusion, (3) knowing one note's sk_note reveals nothing about
other notes' nullifiers.

On-chain, a new output in the state diff contains:

  - note_cm           (32 bytes)    UTXO commitment
  - owner_commitment  (32 bytes)    ownership commitment
  - value_commitment  (32 bytes)    value hiding commitment
  - kem_ciphertext    (1,088 bytes) ML-KEM-768 encapsulation
  - enc_note          (64 bytes)    ChaCha20-Poly1305 encrypted payload
  - view_tag          (2 bytes)     fast-scan filter

Total per output: 1,250 bytes.

This is larger than a naive 96-byte design. The increase is the honest
cost of post-quantum non-interactive receiving. A 96-byte output with
no note discovery mechanism requires mandatory out-of-band communication
for every payment. That is not a blockchain primitive -- it is a
messenger application layered on top. We pay the size cost at L1 and
get genuine post-quantum privacy. Applications that can tolerate a
data-availability layer dependency may optionally move the KEM
ciphertext off-chain, reducing the on-chain footprint to 96 bytes
while fetching ciphertexts from the DA layer. The base protocol
includes ciphertexts on-chain for maximum trustlessness.

For coinbase outputs, the owner_commitment is derived from id_root:

  coinbase_owner = H(id_root || height || "specter.coinbase")

Coinbase outputs have no KEM fields since the miner knows their own
outputs directly:

  Coinbase output: 96 bytes (note_cm + owner_cm + value_cm only).

/**** Note Discovery: Post-Quantum Non-Interactive Receiving ****/

This is the hardest design problem in Specter. For a privacy protocol,
outputs must be identifiable only by the recipient -- without revealing
which address they belong to to external observers.

Classical solutions use ECDH (Monero's stealth addresses, Zcash's
note encryption). Both are broken by Shor's algorithm.

The post-quantum solution is ML-KEM-768 (FIPS 203), based on the
Module Learning with Errors (MLWE) problem. MLWE resists both
classical and quantum attacks. The best known quantum algorithms
achieve only subexponential speedup via BKZ-based lattice reduction,
not polynomial as Shor's algorithm achieves against ECDH.

The construction follows the Module-LWE Stealth Address Protocol
(Mikic, Srbakoski, Praska, ePrint 2025/112):

  Recipient setup (done once, address published):
    scan_key    = H(REV || "specter.scan-key")
    (ek, dk)    = ML-KEM-768.KeyGen(HKDF(scan_key, "specter.kem-seed"))
    address     = ek    [1,184 bytes, published once]

  Sender creates an output addressed to recipient:
    1. (K_shared, C_kem) = ML-KEM-768.Encap(ek_recipient)
       K_shared is 32 bytes. C_kem is 1,088 bytes (published on-chain).
    2. K_note = HKDF(K_shared || "specter.note-key")
    3. view_tag = K_note[0:2]
    4. owner_commitment = H(K_note || "specter.owner")
    5. note = (owner_commitment, value_commitment, rho)
    6. plaintext = encode(v, r, rho)   [48 bytes: value + blinding + rho]
       enc_note = ChaCha20-Poly1305.Encrypt(K_note, plaintext)  [64 bytes]
    7. Publish: (note_cm, owner_cm, value_cm, C_kem, enc_note, view_tag)

  Recipient scans new state diff:
    For each output:
      1. Check view_tag against H(ML-KEM-768.Decap(dk, C_kem))[0:2]
         -- This filters 99.6% of outputs without full decapsulation
      2. On match: K_note = HKDF(ML-KEM-768.Decap(dk, C_kem) || "specter.note-key")
      3. Decrypt enc_note, recover (v, r, rho)
      4. Verify note_cm matches. Store note.

  Recipient spends:
    Derives owner_secret from stored K_note, proves C1-C4 in STARK.
    KEM fields are not needed for spending.

The ML-KEM encapsulation key ek is derived deterministically from
scan_key via HKDF. The recipient can regenerate dk from their REV
alone -- no separate key storage required for wallet recovery.

Scan performance: 720,000 outputs/day at 0.4% tag hit rate = ~2,880
full ML-KEM decapsulations/day at ~100us each = ~0.3 seconds.
Total scan work including tag checks: ~1 second/day.

Why not CSIDH (isogeny-based)? CSIDH-512 provides approximately 70-bit
classical security per recent structural analysis (ePrint 2024/2042).
NIST standardized no isogeny-based KEM (SIDH/SIKE was broken in 2022).
Isogeny operations are 100-1000x slower than ML-KEM. Not production.

Why not hash-only? No such construction exists. Stealth address
discovery requires non-interactive key agreement. Known hash-only
primitives (SPHINCS+, XMSS, Winternitz) are signature schemes with
no key-exchange analog. This is an open research problem.

/**** Why STARKs ****/

I reject trusted setup. This is non-negotiable.

Groth16 requires ceremonies. BN254 curves are not quantum-safe.
These are disqualifying properties for a system meant to last.

STARKs are transparent. No trusted setup. No pairings. Just hash
functions and polynomial commitments via FRI. Under standard
assumptions about collision-resistant hashing, they provide
post-quantum security.

The trade-off: STARK proofs are larger than SNARKs. A single
per-transition proof is 30-80 KB. The aggregated block proof is
50-130 KB. This is larger than a Groth16 proof (~260 bytes) but
Groth16 requires a trusted setup and is not post-quantum. I accept
the size trade-off.

Per-transition proofs never touch the permanent chain. Through
recursive aggregation, hundreds of proofs compress into one block
proof. Through proof pruning, even the block proof is discarded
after 1440 blocks. The permanent record contains only state diffs.

/**** Proving System: Plonky3 over BabyBear ****/

I use Plonky3 as the proving backend with the BabyBear field
(p = 2^31 - 2^27 + 1 = 2,013,265,921). The reasons:

  1. BabyBear is a 31-bit field. Field multiplications take ~0.4
     cycles with SIMD (AVX-512 or NEON). Goldilocks is 64-bit,
     roughly 2-4x slower per operation.

  2. Plonky3 has native GPU support via ICICLE (CUDA, Metal, CPU
     backends). ICICLE-Plonky3 achieves 3-7x speedup over CPU SIMD
     for NTT and constraint evaluation.

  3. The ecosystem is consolidating around BabyBear. SP1 (Succinct),
     RISC Zero, and Polygon Miden all use BabyBear. Production
     infrastructure, not research.

  4. Plonky3 supports recursive composition via plonky3-recursion.
     SP1's production recursive pipeline runs entirely on Plonky3
     over BabyBear. This is deployed infrastructure.

  5. BabyBear requires a degree-4 (quartic) extension for 128-bit
     security. Most prover work stays in the 31-bit base field.

  6. BabyBear's two-adicity is 2^27, supporting NTT domains of up
     to 134 million rows. Sufficient for any transition trace or
     the recursive aggregation circuit.

On Poseidon2 post-quantum security: current parameters target 128-bit
classical security. Quantum security is approximately 64 bits (Grover).
For the ZK circuit binding property this is sufficient. User secret
hashing (REV and scan_key derivation) uses BLAKE3, which provides
128-bit post-quantum security.

Stwo (StarkWare's Circle STARK prover over M31 / Mersenne-31) is a
candidate backend for future versions. Stwo achieves 940x throughput
improvement over the previous Stone prover and is live on Starknet
mainnet. BabyBear/Plonky3 remains the choice due to superior
recursive composition tooling.

The performance implications are concrete. On an Apple M3 Pro, Plonky3
proves ~1.7-2M Poseidon2 hashes per second. A Specter transition proof
requires roughly 100-140 Poseidon2 evaluations for ownership hashes,
nullifier derivations, Merkle paths, and value commitments. Proof
generation targets 2-5 seconds on a modern multi-core CPU.

/**** A Note on Zero Knowledge ****/

Standard STARK constructions provide computational integrity but not
zero knowledge by default. The proof may leak information about the
witness. For Specter, this matters because the witness contains
owner_secret and value data that must not be revealed.

I achieve ZK through the rigorous masking technique formalized by
Habook and Al Kindi (ePrint 2024/1037):

First, witness masking: for each trace column polynomial, a random
low-degree polynomial is added that vanishes on the trace domain H.
This randomizes polynomial evaluations outside H without affecting
constraint satisfaction within H.

Second, DEEP composition polynomial masking: an additional random
low-degree polynomial is added to the final DEEP composition
polynomial, preventing information leakage through the constraint
evaluation channel.

The masking coefficient count h satisfies:

  h >= 2 * S * (e * n_D + n_F) + n_F

Where S=2 (quotient splitting), e=4 (BabyBear extension degree),
n_D=1 (DEEP queries), n_F~50 (FRI queries). This yields h ~ 500
coefficients. Overhead: under 5% proving time, negligible proof size.

With both masking layers, the underlying IOP achieves honest-verifier
zero knowledge (HVZK). Fiat-Shamir upgrades this to full computational
zero knowledge in the Random Oracle Model. The ZK property composes
through recursive aggregation: the aggregator sees only proofs and
their public inputs, never witnesses.

/**** Transition Authorization ****/

The fundamental unit in Specter is the state transition, not the
transaction. A state transition is:

  StateTransition:
    consumed_nullifiers[]  (each 32 bytes)
    created_outputs[]      (each 1,250 bytes)
    fee                    (8 bytes)
    pre_state_root         (32 bytes)

The authorization digest binding the STARK proof to a specific
transition:

  transition_hash = H(
    "specter.transition"      ||
    pre_state_root            ||
    consumed_nullifiers[]     ||
    created_outputs[].note_cm ||
    fee
  )

  Auth = H(auth_root || transition_hash || "specter.auth")

This replaces v3's tx_hash construction. The critical addition is
pre_state_root: the proof is bound to a specific chain state. A proof
generated against state root A cannot be replayed against state root B.
This closes the replay attack surface present in v3.

What the proof covers: the prover knows auth_root such that Auth is
correctly derived; Auth binds to transition_hash which binds to
pre_state_root, all nullifiers, all output commitments, and fee.
Nothing is mutable by relays. All transition fields are covered by
transition_hash and thus by the STARK proof.

What is not covered: the STARK proof itself (the proof attests to the
digest, not vice versa). Block-level fields (height, timestamp, PoW
nonce) belong to the block header.

/**** User-Side Proving ****/

Each user generates their own STARK proof when creating a state
transition. This is the critical architectural decision that
distinguishes Specter from systems where block producers prove.

When a user creates a transition, their wallet:

  1. Fetches MMR inclusion proofs for input notes from a full node
  2. Fetches nullifier non-membership witnesses from a full node
  3. Builds an execution trace for the state transition
  4. Includes all constraints (C1-C4, V2-V4)
  5. Applies witness and DEEP composition masking
  6. Generates a STARK proof (~30-80 KB)
  7. Discards the witness data (owner_secret, v, r, blinding factors)
  8. Broadcasts the proof fragment to the network

The proof fragment published to the network:

  - consumed_nullifiers[]  (public: for double-spend detection)
  - created_outputs[]      (public: note_cm, owner_cm, value_cm,
                            kem_ciphertext, enc_note, view_tag)
  - fee                    (public)
  - pre_state_root         (public: the chain state this proof is for)
  - STARK proof            (~30-80 KB, discardable after aggregation)

No witness data. No private information. The REV never leaves the
user's device.

The proof statement covers C1-C4 and V2-V4. Input existence (V1) and
state-root consistency are verified outside the STARK via MMR inclusion
proofs and nullifier predecessor proofs checked by the consensus layer.

Proof generation targets:
  - 2-5 seconds on modern multi-core CPU (Plonky3 over BabyBear)
  - 1-3 seconds on GPU via ICICLE
  - Sub-second possible as hardware and proving optimizations mature

For low-powered devices, proof generation can be delegated to a trusted
prover service. This requires sharing witness data with that service.
The privacy trade-off is the user's to make.

/**** Recursive Proof Aggregation ****/

This is the mechanism that transforms Specter from a system where every
node verifies hundreds of individual proofs per block into a system
where every node verifies exactly one.

The aggregation circuit proves not just that each state transition is valid, 
but that the sequence of transitions correctly evolves the state root from 
pre_state_root to post_state_root. This closes the v3 security gap where a 
malicious block producer could include valid proofs but commit a false
post_state_root (e.g., inflating balances).

The aggregation circuit proves:

  state_root_0 = pre_state_root

  for i in range(N):
    verify(proof_i, public_inputs_i)
    assert public_inputs_i.pre_state_root == state_root_{i-1}
    state_root_i = apply_nullifiers(state_root_{i-1}, nullifiers_i)
    state_root_i = apply_outputs(state_root_i, outputs_i)

  assert state_root_N == post_state_root

The apply functions compute MMR and IMT root updates in-circuit via
Poseidon2. This adds approximately 45,000 in-circuit Poseidon2
evaluations per block (500 transitions x 30 MMR levels x 2 operations
+ 1,000 nullifiers x 30 IMT levels). At ~2M Poseidon2/second on GPU,
this adds ~1.5 seconds to aggregation time.

Pipeline (binary tree recursion):
  Level 0: N individual per-transition proofs (user-generated)
  Level 1: N/2 pairwise recursive proofs
  ...
  Level log2(N): 1 block proof covering entire state evolution

The final aggregated proof is 50-130 KB regardless of N. It proves
that starting from a known pre_state_root, applying the state diff
correctly produces post_state_root.

Trust model: the aggregator never sees witness data from inner proofs.
ZK composes through recursive verification: the aggregated proof
reveals nothing about individual transition witnesses.

The block time of 120 seconds accommodates GPU-assisted aggregation
at 30-90 seconds, leaving 30-90 seconds for PoW search and propagation.

Performance targets for the block producer:
  - Aggregation of 500 proofs: 30-90 seconds on GPU via ICICLE
  - Output proof size: 50-130 KB (constant in transition count)
  - Verifier time for aggregated proof: 5-15 ms

/**** Block Structure ****/

A Specter block is a proof of state evolution. It contains no
transaction objects.

  Block:
    header:
      version               (4 bytes)
      height                (8 bytes)
      timestamp             (8 bytes)
      prev_hash             (32 bytes)
      pre_state_root        (32 bytes)   UTXO MMR root before this block
      post_state_root       (32 bytes)   UTXO MMR root after this block
      nullifier_root        (32 bytes)   nullifier IMT root after this block
      diff_root             (32 bytes)   Merkle root of state_diff
      aggregated_proof_hash (32 bytes)
      nonce                 (8 bytes)
      difficulty_target     (4 bytes)
      anchor_height         (8 bytes)
    ───────────────────────────────────
    Header total: ~200 bytes

    aggregated_proof     (50-130 KB, prunable after 1440 blocks)

    state_diff:
      new_nullifiers[]   (32 bytes each)
      new_outputs[]      (1,250 bytes each)
      fee_kernels[]      (32 bytes each, fee + coinbase attribution)

Key distinction from v3: tx_root is replaced by diff_root. There are
no transaction objects. The state_diff is a flat list of nullifiers
consumed, outputs created, and fees paid. Individual user transitions
are merged at the aggregation step. The mapping from "which nullifiers
were consumed together" is not preserved in the block, providing
structural unlinkability at the L1 layer.

Verifying a block:

  1. Check PoW: BLAKE3(BLAKE3(header)) < difficulty_target
  2. Check ASERT difficulty against parent
  3. Verify aggregated_proof against public inputs:
     (pre_state_root, post_state_root, nullifier_root, diff_root)
  4. Apply state_diff to local state (MMR appends, IMT insertions)
  5. Verify locally computed post-state roots match header
  6. Verify coinbase output = block_reward + sum(fees)

Step 3 is one proof verification: 5-15 ms. Block validation targets
under 1 second for 500-transition blocks.

/**** The Mempool: Proof Fragment Pool ****/

The Specter mempool contains proof fragments, not transactions.

  ProofFragment:
    consumed_nullifiers[]  (public)
    created_outputs[]      (public)
    fee                    (public)
    pre_state_root         (32 bytes)
    stark_proof            (~30-80 KB)

Nodes admit a fragment to the mempool after:

  Stage 1 (nanoseconds):  structural checks, size bounds, encoding validity
  Stage 2 (microseconds): duplicate nullifier check, fee minimum,
                          nullifier set non-membership
  Stage 3 (1-5 ms):       STARK proof verification

This cost-escalating ladder ensures cheap garbage cannot force expensive
proof verification. Peers that repeatedly fail Stage 1 or 2 are
deprioritized.

Conflict detection: two fragments conflict if they share any consumed
nullifier. Conflict resolution: the higher-fee fragment is preferred;
ties broken by arrival time. A fragment generated against an outdated
pre_state_root is rejected at Stage 1 after a block advance.

Fee market: fees are per-fragment. The natural unit is fee per nullifier
consumed (each consumed nullifier represents one UTXO spent). The block
producer selects a conflict-free fragment set maximizing total fee.

Fragment privacy: fragments broadcast individually reveal the association
between consumed nullifiers and created outputs before block production.
After aggregation this association is not preserved in the block.
Network-layer privacy (Dandelion++ routing, Tor) is orthogonal to L1
design and strongly recommended for wallet software.

/**** Multi-Input Unlinkability ****/

When a user consumes multiple notes in a single transition, all
nullifiers appear together, revealing they are controlled by the same
spender. This is the common-input-ownership linkage Bitcoin also
exhibits.

Mitigation: each note has its own note-local owner_secret (not the
master REV). The proof demonstrates knowledge of owner_secret for each
input independently. An observer sees multiple nullifiers but cannot
determine whether they come from the same master identity or multiple
cooperating parties.

Wallet policy: coin selection minimizes multi-input transitions.

Perfect unlinkability requires dummy inputs (Zcash Orchard action
model): each proof contains exactly one consume and one create, with
dummy actions indistinguishable from real ones. This is a future
extension.

/**** Consensus Validation Semantics ****/

All state transitions within a block are validated against the block's
pre_state_root (the parent block's post_state_root). Two transitions
within the same block cannot consume the same nullifier. Outputs created
in a block are spendable in the next block.

The aggregated proof encodes the application order of transitions within
a block. This ordering is determined by the block producer. The choice
of ordering does not affect the final post_state_root, provided there
are no nullifier conflicts -- which the proof enforces. Block producers
have ordering discretion analogous to miner discretion in Bitcoin, but
cannot affect the cryptographic state result.

/**** Value Commitments ****/

I use hash-based commitments instead of Pedersen commitments:

  value_commitment = H_zk(v || r || "specter.value")

Where v is the 64-bit value, r is the 256-bit blinding factor, and
the domain tag provides separation. Computationally hiding and binding
under collision resistance. Post-quantum safe.

The cost: no additive homomorphism. Balance verification happens inside
the STARK proof. The prover demonstrates in zero knowledge that input
and output values satisfy conservation. Range proofs (v in [0, 2^64))
are decomposed into bits inside the trace. This is ~50-100 additional
constraints per value, negligible against the full circuit.

The result: no elliptic curves in the authorization path. No pairings.
Post-quantum security of spend authorization is entirely hash-based.

/**** The Nullifier Set ****/

Each spent note produces a unique, unlinkable nullifier:

  sk_note   = H(null_root || rho || "specter.note-null")
  nullifier = H(sk_note || note_cm || "specter.nullifier")

The nullifier is deterministic: the same output always produces the
same nullifier when spent by the same owner. An observer cannot compute
the nullifier without knowing null_root. Only the owner knows the
mapping from commitment to nullifier.

Nullifiers are stored in an Indexed Merkle Tree (IMT). When spending,
the user proves inside their STARK proof that the nullifier is correctly
derived AND does not already exist in the tree (non-membership via the
low-nullifier predecessor proof).

Why IMT instead of sparse Merkle trees? A sparse Merkle tree with
256-bit keys requires 256 hash operations per non-membership proof
inside a ZK circuit. An IMT uses a sorted linked-list structure where
non-membership is proven by showing a "low nullifier" (value less than
target, next-pointer greater than target). This requires only a
membership proof at tree depth ~32 plus a range check. Roughly 8x
fewer hashes. Aztec and Nomos have independently arrived at the same
conclusion.

The nullifier set is permanent. It cannot be pruned without breaking
double-spend protection. Growth rate: 1,000 nullifiers per block x
720 blocks/day x 32 bytes = 22 MB/day. At 10 years: ~80 GB for the
raw nullifier set. This is the dominant long-term storage cost for full
nodes and there is no clean solution with pure hash-based cryptography.

RSA accumulators achieve O(1) non-membership proofs but require a
trusted modulus or class group assumptions incompatible with the
hash-only goal. Lattice-based accumulators are an active research area
with no mature production construction. Nullifier set compression via
novel post-quantum authenticated data structures is the most important
open research problem for Specter's long-term scaling. We name it
explicitly rather than hand-wave it.

/**** Cut-Through ****/

If a note is created and consumed within the same aggregation batch,
both the output and nullifier can be removed from the state diff:

  Before: A spent to create B; B spent to create C
  State diff: [A consumed, C created]    (B never touches the chain)

Cut-through is applied by the block producer before block assembly. It
is an optimization, not a consensus rule. Full nodes that wish to
reconstruct the full in-batch history must retain pre-aggregation proof
fragments. After the pruning window, this is archival data.

Cross-block cut-through is applied during pruning (after 1440 blocks):
if an output was created in block N and consumed in block N+k (both
within the pruning window), both the output commitment and nullifier
can be eliminated from the pruned permanent record. Only the net
unspent set and all nullifiers from consumed outputs not also created
in the same pruning batch remain.

The permanent chain state converges toward:

  - Block headers (with state root commitments)
  - Unspent output commitments (UTXO set)
  - All nullifiers (permanent, cannot prune)
  - Fee kernels

/**** Chain Growth ****/

Chain growth is managed through four complementary mechanisms:

  1. Recursive Aggregation: reduces proof data 100-250x per block
  2. Proof Pruning: eliminates all proof data from permanent storage
     after 1440 blocks (~48 hours)
  3. Cut-Through: removes intermediate outputs
  4. MMR Accumulator: compact state commitment

Per block (500 transitions, 2-in/2-out, after proof pruning):

  New nullifiers:  1,000 x 32 =         32,000 bytes
  New outputs:     1,000 x 1,250 =   1,250,000 bytes
  Fee kernels:       500 x 32 =         16,000 bytes
  Header:                               ~200 bytes
  ──────────────────────────────────────────────────
  Per-block permanent:               ~1,298 KB

Before proof pruning: add ~90 KB aggregated proof per block.

Daily chain growth (permanent, post-pruning): ~1,298 KB x 720 = ~937 MB/day.

This is larger than v3's 110 MB/day. The increase is entirely the
ML-KEM ciphertext (1,088 of 1,250 bytes per output). This is the
honest cost of post-quantum non-interactive receiving.

Protocol comparison (permanent chain growth after proof pruning):

  Protocol                   Per-day    Note discovery?   PQ?
  ──────────────────────────────────────────────────────────────
  Bitcoin                    ~250 MB    N/A               No
  Monero                     ~150 MB    Yes (ECDH)        No
  Zcash Orchard              ~180 MB    Yes (ECDH)        No
  Specter                    ~937 MB    Yes (ML-KEM)      Yes

An honest table. Specter v3's 110 MB/day claim was for an incomplete
system that required out-of-band communication for every payment. The cost of 
post-quantum non-interactive privacy is real.

Full node storage at year 5:

  Nullifier set (raw + IMT overhead): ~120 GB
  UTXO set (MMR):                     ~15-25 GB
  Block headers:                      ~1 GB
  Chain data (state diffs):           ~1,700 GB
  ──────────────────────────────────────────────
  Total:                              ~1,850 GB

This is larger than a modern Bitcoin full node (~600 GB in 2025) due
to the per-output KEM ciphertext. A DA-layer variant (KEM ciphertexts
off-chain, 96-byte on-chain outputs) reduces this to ~260 GB at year 5,
comparable to Bitcoin, at the cost of DA layer availability assumptions.

Future path to smaller chain:

  1. ML-KEM-512 (lower security margin): 768-byte ciphertext reduces
     per-output to ~930 bytes, chain growth to ~700 MB/day.
  2. DA-layer KEM variant: on-chain output drops to 96 bytes (~110 MB/day),
     KEM ciphertexts served from a data availability layer.
  3. Breakthrough in hash-only non-interactive note discovery: would
     eliminate the KEM entirely. No such construction currently known.

/**** Stateless Validation ****/

Specter uses a Merkle Mountain Range (MMR) for the UTXO set, following
Utreexo (Dryja, MIT DCI, 2019). The MMR root provides a compact
commitment to the entire UTXO set.

There is one type of full node. It maintains:

  - Block headers chain (complete)
  - Full MMR UTXO set
  - Full nullifier IMT
  - Recent blocks with aggregated proofs (within 1440-block window)
  - Pruned block data (state diffs without proofs, permanent)

A full node can verify any block, serve MMR inclusion proofs and
nullifier non-membership witnesses to wallets.

Light clients maintain block headers and state root commitments only.
They verify the aggregated proof per block (5-15 ms, ~1.3 KB state).
They cannot generate proofs without fetching witnesses from a full node.

Archive behavior (retaining aggregated proofs past the pruning window)
is a local configuration choice, not a protocol type. The protocol
does not require or reward it.

When spending, the wallet includes a short MMR inclusion proof (~500
bytes) alongside the proof fragment, outside the STARK proof. The
verifier checks membership separately. This avoids the proving cost
of embedding full Merkle path verification inside the arithmetic
circuit.

Compact nodes: block headers + MMR roots + nullifier roots. Less than
1.3 KB of state. Verify one aggregated proof per block. Cannot generate
spending proofs without fetching witnesses from peers.

New nodes syncing from genesis have two options:

  1. Full sync: download all blocks including aggregated proofs
     (from nodes retaining them), verify every block proof, then prune.

  2. Checkpoint sync: download a verified UTXO set snapshot at a
     recent checkpoint, download block headers from genesis to verify
     the PoW chain, then sync recent blocks with proofs. Analogous to
     Bitcoin's assumeUTXO.

Historical validity after the pruning window becomes an archival
availability assumption: a new node syncing after 1440 blocks cannot
independently verify pruned proofs without obtaining them from a node
that retained them. This is the same trust model as Bitcoin's
assumeUTXO, applied to proof data.

/**** Concrete Hash Function: Poseidon2 over BabyBear ****/

Throughout this paper, H_zk denotes Poseidon2 over the BabyBear field
(p = 2^31 - 2^27 + 1). Poseidon2 was chosen for three reasons:

  1. ZK-friendly: ~126 constraints per invocation inside a STARK
     over BabyBear (width 16, rate 8, 8 external rounds, 13 internal
     rounds, S-box x^7). Up to 4x fewer constraints than original
     Poseidon.

  2. Security: 128-bit classical collision resistance. Active
     cryptanalysis (ePrint 2025/937, 2025/950, 2026/150) has found
     attacks only on round-reduced instances. Full-round Poseidon2
     over BabyBear remains unbroken as of March 2026.

  3. Performance: 31-bit field multiplications are ~4x cheaper than
     64-bit (Goldilocks) on standard hardware.

For Merkle tree commitments during proving, BLAKE3 is used for upper
tree levels (not re-proved inside the STARK). Poseidon2 is used for
leaf-level hashing (proved inside the STARK). This hybrid approach
reduces commitment time by approximately 50%.

For all non-ZK uses (block header hashing, key derivation, PoW),
BLAKE3 is used for its native speed (~3-8 GB/s single-threaded) and
128-bit post-quantum preimage resistance.

/**** Consensus ****/

Proof-of-work using double BLAKE3:

  pow_hash = BLAKE3(BLAKE3(header_bytes || nonce))

Valid if pow_hash < difficulty_target. BLAKE3 is already in Specter's
dependency tree. Verification takes nanoseconds.

The spam rate-limiter in Specter is not PoW. It is proof generation
cost: 2-5 seconds of CPU per transition. PoW provides block-level
Sybil resistance and emission scheduling only.

Block time: 120 seconds. Accommodates GPU-assisted proof aggregation
(30-90 seconds) plus PoW search and propagation.

At 500 transitions per block, sustained throughput is ~4 transitions
per second.

Difficulty adjustment: ASERT (aserti3-2d), deployed on Bitcoin Cash
since November 2020, proven mathematically to converge:

  next_target = anchor_target * 2^((time_delta - 120 * (height_delta + 1)) / halflife)

With halflife = 172,800 seconds (2 days). Fixed-point integer
arithmetic to avoid floating-point non-determinism.

/**** Emission Schedule ****/

Initial block reward: 50 SPEC.

Smooth exponential decay with perpetual tail emission:

  reward(height) = max(floor(50 * 0.999999 ^ height), 0.6)

Floor of 0.6 SPEC/block reached after ~4.4 million blocks (~16.8 years
at 720 blocks/day). Tail emission continues indefinitely.

Key properties:
  - Total pre-tail supply: ~49.4 million SPEC
  - Tail emission: 0.6 SPEC/block = ~158,000 SPEC/year
  - Year 10 inflation: ~1.8%
  - Year 20 inflation: ~0.3%
  - Approaches 0% asymptotically

The tail emission ensures miners always have baseline revenue
independent of fee markets. The smooth decay avoids halving shocks
that create speculative cycles.

No premine. No founder allocation. No ICO.

/**** Security Goals ****/

Specter provides:

  1. Signatureless authorization: no signatures, no signing public keys
     on-chain. Authorization is a ZK proof of secret knowledge.
  2. Post-quantum spend soundness: under collision resistance of BLAKE3
     and soundness of the Plonky3 STARK, no quantum adversary can forge
     a valid spend proof. No elliptic curve or lattice assumption in
     the authorization path.
  3. Post-quantum note confidentiality: values and note contents are
     hidden under hash-based commitments and ChaCha20-Poly1305.
  4. Non-interactive receiving: senders pay to published ML-KEM
     addresses without any recipient interaction.
  5. Post-quantum note discovery: ML-KEM-768 (FIPS 203, MLWE) provides
     128-bit post-quantum security for the note discovery mechanism.
  6. Nullifier unlinkability: an observer cannot determine which UTXO
     was consumed by a given nullifier without knowing null_root.
  7. Structural unlinkability at L1: the block state_diff does not
     preserve the association between consumed notes and created notes.
     Individual transitions are merged at aggregation.
  8. O(1) block verification: one STARK proof per block, independent
     of transition count.
  9. State root integrity: the aggregated proof cryptographically
     enforces that post_state_root is correctly derived from
     pre_state_root by applying the state_diff. A malicious block
     producer cannot commit a false state root and produce a valid proof.

Specter does NOT provide at L1:

  1. Network-layer anonymity: requires Tor, Dandelion++, or equivalent.
  2. Mempool origin hiding: proof fragments reveal nullifiers and outputs
     before aggregation to nodes receiving them.
  3. Multi-transition unlinkability in the mempool: fragments submitted
     individually can be correlated by a mempool-observing adversary.
  4. Recovery privacy after REV compromise: a compromised REV allows
     retroactive identification of all outputs.
  5. Protection against a malicious proving service: delegated proving
     requires sharing witness data.
  6. Perfect multi-input unlinkability: multiple consumed nullifiers in
     a single proof create a same-spender statistical signal. Dummy
     inputs are a future extension.

/**** What I Achieve ****/

No signatures on-chain. Ever.

No signing public keys revealed. No key used for spending verification
is ever exposed.

No elliptic curves in the authorization or spend proof path. Post-quantum
spend security by construction, not by retrofit.

Post-quantum note discovery via ML-KEM-768. Senders pay without
recipient interaction. Recipients scan and discover notes. No out-of-band
communication required for any payment.

No trusted setup. Transparent STARK proofs only.

One proof per block via recursive aggregation. Block verification in
5-15 ms regardless of transition count.

State root integrity enforced cryptographically: the aggregated proof
proves correct state evolution, not just transition validity.

Proof pruning after 1440 blocks.

Cut-through eliminating intermediate outputs from the permanent record.

UTXO accumulator (MMR) enabling compact nodes with ~1.3 KB state.

Fair emission. No premine. Smooth decay. Perpetual tail reward.

ASERT difficulty adjustment with mathematical convergence guarantees.

/**** Questions I Woke Up Sweating About ****/

Q: Why ML-KEM if everything else is hash-based?

A: Because hash-based non-interactive note discovery does not exist.
   No construction is known that allows a recipient to identify outputs
   without either an asymmetric primitive or per-payment out-of-band
   communication. We refuse mandatory out-of-band communication -- that
   is not a blockchain, that is a messenger. ML-KEM is standardized by
   NIST (FIPS 203, August 2024), provides 128-bit post-quantum security,
   and is being deployed in TLS 1.3. It is the minimum necessary
   asymmetric component.

Q: The KEM ciphertext is 1,088 bytes. That is large.

A: It is. This is the honest cost of post-quantum non-interactive
   privacy. X25519 ephemeral key is 32 bytes -- and Shor's algorithm
   breaks X25519. There is no free lunch. The alternative is classical
   ECDH (not quantum-safe) or mandatory online receivers (not a payment
   system). We choose correctness. The DA-layer variant reduces this
   cost for applications that can tolerate the tradeoff.

Q: Why not CSIDH for stealth addresses?

A: CSIDH-512 provides approximately 70-bit classical security per
   recent structural attacks (ePrint 2024/2042). NIST standardized no
   isogeny-based KEM -- SIDH/SIKE was broken completely in 2022.
   Isogeny operations are 100-1000x slower than ML-KEM. Not production.

Q: What about Poseidon2 post-quantum security?

A: Production parameters (width-16, rate-8, 8+13 rounds, BabyBear)
   achieve 128-bit classical security. Recent attacks (ePrint 2025/937,
   2025/950) found vulnerabilities only on round-reduced instances.
   Full-round Poseidon2 is unbroken. Quantum security is ~64 bits
   (Grover), sufficient for ZK circuit binding. User secret hashing
   uses BLAKE3 with 128-bit post-quantum security.

Q: What if someone loses their REV?

A: They lose their funds. Same as losing a Bitcoin private key.
   Social recovery schemes are a future application layer concern.

Q: How does the receiver scan?

A: The 2-byte view_tag filters 99.6% of outputs before full
   decapsulation. At 720,000 outputs/day, full ML-KEM decapsulation
   runs on only ~2,880 outputs/day (~0.3 seconds). Total scan work
   is ~1 second/day on consumer hardware.

Q: Is Poseidon2 over BabyBear post-quantum at 128 bits?

A: No. Current parameters target 128-bit classical security.
   Quantum security is approximately 64 bits. Sufficient for ZK
   circuit binding. User secret hashing uses BLAKE3 (128-bit PQ).
   Future versions may adopt doubled-round Poseidon2 if formal
   128-bit post-quantum analysis is established.

Q: What is the chain size at 10 years (DA-layer variant)?

A: With KEM ciphertexts on a DA layer (96-byte on-chain outputs):
   Nullifier set: ~160 GB. Chain data: ~300 GB. Total: ~500 GB.
   Without DA layer: ~3,500 GB. The DA variant is the practical path
   for long-term sustainability while retaining PQ note discovery.

Q: Doesn't recursive aggregation centralize block production?

A: It raises the hardware floor. Aggregating 500 proofs takes 30-90
   seconds on GPU, comparable to GPU requirements for competitive PoW
   mining. Aggregation is deterministic and verifiable -- a miner
   cannot produce an invalid aggregated proof. A miner who omits valid
   fragments can be detected by comparing the mempool to the block.

Q: Can the block producer learn private information during aggregation?

A: No. Recursive verification is a proof-of-proofs construction. The
   aggregator receives only proofs and their public inputs (nullifiers,
   commitments, fee). Inner proofs are zero-knowledge. ZK composes
   through recursion.

Q: What is lost without transactions at L1?

A: Transaction IDs, transaction history in block explorers, per-sender
   fee attribution, causal traces. These are replaced by: state root
   commitments, aggregated validity proofs, and wallet-side record-
   keeping. The privacy gain is real: no transaction graph exists at L1.

Q: What about scripts, hashlocks, timelocks?

A: Constraints inside the STARK circuit. A hashlock adds:
   H_zk(preimage) == hash. A timelock adds: block_height >= t as a
   public input constraint. No scripting language -- a constraint
   system. Simple conditions can be added without significant proof
   size increase.

Q: How does the sender communicate value to the receiver?

A: Value and blinding factor are encrypted inside enc_note (the
   ChaCha20-Poly1305 payload). The receiver decrypts this when they
   scan. No out-of-band communication needed. The encrypted payload
   (64 bytes on-chain) carries exactly: value (8 bytes), blinding
   factor (32 bytes), rho (8 bytes), padded to 48 bytes plaintext.

/**** Future Directions ****/

DA-layer KEM variant. Moving KEM ciphertexts to a data availability
layer reduces on-chain footprint to 96 bytes per output. Recipients
fetch ciphertexts from the DA layer. Recommended for production
deployments prioritizing chain size over full L1 trustlessness.

Dummy transitions for multi-input unlinkability. Each proof contains
exactly one consume and one create, with dummy actions indistinguishable
from real ones. Eliminates same-spender statistical linkage.

ML-KEM-512 variant. Ciphertext shrinks from 1,088 to 768 bytes
(100-bit post-quantum vs. 130-bit for ML-KEM-768). Per-output: ~930
bytes. Chain growth: ~700 MB/day.

WHIR-based proving. WHIR (EUROCRYPT 2025) achieves 1.25-2.46x smaller
proofs than FRI-based STARKs. Research-only as of 2026; candidate for
v5 proving backend.

Circle STARKs (Stwo over M31). StarkWare's Stwo prover achieves 940x
throughput improvement over previous generation and sub-second recursive
proving. Live on Starknet mainnet. Candidate backend upgrade once
recursive composition tooling reaches Plonky3's level.

Proof markets. Users outsource proving to competing provers. Privacy
trade-off: the prover sees the witness. Trusted execution environments
or threshold proving mitigate this.

State channels. Off-chain transitions with on-chain settlement proofs.

STARK-to-SNARK wrapping. For cross-chain bridges requiring constant-
size verification, wrap the aggregated STARK in a Groth16 SNARK (~260
bytes). Introduces trusted setup at the bridge layer only. L1 remains
fully transparent.

Formal verification of the AIR constraints. Machine-checked proof that
the constraint system correctly enforces the intended transition
semantics.

Throughput scaling via aggregation depth. As recursive proving hardware
improves, block size can increase without increasing verification cost
(verification is O(1)). Block time and transition count become
parameters tunable against aggregation latency.

/**** Open Research Problems ****/

The following are unsolved and explicitly named:

  1. Hash-only post-quantum non-interactive note discovery. No
     construction is known. ML-KEM is the current solution; a hash-only
     replacement would be a major cryptographic contribution.

  2. Succinct post-quantum non-membership proofs for dynamic sets.
     Enabling light clients to verify double-spend protection without
     storing the full nullifier set, using only hash primitives.

  3. Formal post-quantum security proof for Poseidon2 over BabyBear
     at 128-bit quantum security. Current analysis covers 128-bit
     classical security only.

  4. Privacy-preserving fee markets for proof fragment mempools.
     Preventing fee-based deanonymization while maintaining economic
     incentives.

  5. Formal game-based privacy definition for transactionless blockchains.
     What exactly is proven about privacy when no transaction objects
     exist, and against what adversary class?

/**** Conclusion ****/

Bitcoin gave us decentralized money. Specter eliminates the weakest
link: the signature. Then eliminates transactions entirely from the
base layer.

A signature says "I possess the key that matches this public key."
A Specter proof says "The state evolved correctly and I am authorized."
Nothing more. The chain sees only state roots, commitments, and one
proof per block. Identity is spectral -- present but invisible.

The proofs are temporary. The state transitions are permanent. After
pruning, the chain carries permanent output data (including ML-KEM
ciphertexts for note discovery) and nullifiers. It is larger per output
than a design without note discovery -- because a design without note
discovery is not a complete payment system.

This is not a chain of transactions. It is a chain of state evolution
proofs. Authorization is a zero-knowledge statement about knowledge.
Every block is one proof that the state is valid, verified in
milliseconds by every node. The KEM ciphertexts are the price of
post-quantum privacy. We pay it.

Five derived roots. One ML-KEM key pair. One STARK per block. Zero
signatures. Zero transactions. The design is deliberately minimal in
its trust assumptions, deliberately complete in its functionality.

The specter is already here. You just cannot see it.
