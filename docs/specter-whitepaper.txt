SPECTER

Ignotus Nemo

Version 2.0
26 March 2026


/**** Abstract ****/

Specter is a transparent proof-of-work validity chain that permanently
removes public-key signatures from the authorization layer.

Standard cryptocurrency chains authorize spends by storing signature
artifacts permanently on-chain. Every historical signature is retained
forever, including the public keys that could be attacked by a
sufficiently large quantum computer. The naive post-quantum "fix" —
replacing ECDSA with a lattice-based signature — leaves the same
structural problem: large, permanent, public authorization material
in canonical chain history.

Specter takes a different approach.

Authorization in Specter is a zero-knowledge proof that the spender
knows a secret preimage bound to the spent output and to the specific
state transition being proposed. No public key is revealed. No
signature is stored. No authorization artifact is retained permanently
in the canonical chain.

The permanent canonical chain stores only state:

  - output records (transparent: address, value, randomness)
  - nullifier records (spentness evidence, 32 bytes each)
  - block headers with cryptographic state commitments
  - fee attribution

Authorization evidence exists only during a bounded proof-retention
window sufficient for consensus verification. After that window,
per-block proofs are replaced by compact epoch validity certificates
that preserve trustless historical verifiability without indefinite
proof storage.

The protocol introduces eight architectural components that together
make this design workable:

  1. Hash-derived addresses with no algebraic public-key structure
  2. Nullifier-based spentness with two-stage BLAKE3 derivation
  3. Anchor-window binding for practical mempool economics
  4. Epoch compression proofs for trustless historical verification
  5. Two-role block production: block producers and miners
  6. Two-phase spend path for wallet UX
  7. Witness bundle protocol with block-producer serving obligations
  8. Proof-aware fee market with consensus-enforced fee distribution

Specter is post-quantum-targeting. Its permanent authorization surface
contains no structures vulnerable to Shor's algorithm. Its security
under the quantum random oracle model rests on the Fiat-Shamir security
of FRI-based STARKs, which was formally established for multi-round
IOPs by Block et al. (ASIACRYPT 2023).


/**** The Problem ****/

Every major cryptocurrency carries a structural liability.

Bitcoin uses secp256k1 elliptic-curve cryptography. Ethereum uses it.
Every major chain uses either elliptic-curve discrete-log signatures or
another public-key construction whose security reduces to a problem
that Shor's algorithm solves in polynomial time on a sufficiently
large quantum computer.

NIST began post-quantum standardization in 2016. In August 2024 it
finalized three standards: FIPS 203 (ML-KEM), FIPS 204 (ML-DSA /
Dilithium), and FIPS 205 (SLH-DSA / SPHINCS+).

The naive answer is: replace ECDSA with ML-DSA or SLH-DSA.

That is wrong. Not because those standards are bad, but because they
do not address the structural problem.

The structural problem is permanent public authorization retention.

Every chain that stores signatures stores them forever. A public key
exposed years before a quantum computer exists is still exposed when
that computer arrives. A signature scheme swap makes future signatures
safer but does nothing for historical authorization material already
permanently embedded in chain history.

The size problem amplifies this. Post-quantum signature schemes are
substantially larger than ECDSA.

FIPS 204 (ML-DSA) signature sizes:

  ML-DSA-44   2,420 bytes   NIST security level 2
  ML-DSA-65   3,309 bytes   NIST security level 3
  ML-DSA-87   4,627 bytes   NIST security level 5

FIPS 205 (SLH-DSA) signature sizes (selected):

  SLH-DSA-SHA2-128s    7,856 bytes   level 1, slow variant
  SLH-DSA-SHA2-128f   17,088 bytes   level 1, fast variant
  SLH-DSA-SHA2-192f   35,664 bytes   level 3, fast variant
  SLH-DSA-SHA2-256f   49,856 bytes   level 5, fast variant

At Bitcoin-like throughput (1,000 spends per block, 720 blocks per
day), permanent authorization retention dominates chain storage:

  ML-DSA-65:          3,309 x 1,000 x 720 = ~2,380 MB/day
  SLH-DSA-SHA2-192f: 35,664 x 1,000 x 720 = ~25,700 MB/day

This is a first-order scaling problem, not an implementation nuisance.
Replacing ECDSA with a post-quantum signature makes the storage
problem much worse, not better.

The correct question is not which signature to use.

The correct question is: can a chain avoid permanent public-key
authorization artifacts entirely?

Specter answers: yes, but only by redesigning the authorization model,
not the signature scheme.


/**** What Specter Is ****/

Specter is a transparent, signatureless, validity-based proof-of-work
chain with ephemeral per-block authorization and epoch-compressed
historical validity.

Transparent

All output values are public. All addresses are public. Supply is
publicly auditable. There is no amount privacy, no address privacy,
and no hidden state.

Signatureless

The protocol layer uses no public-key signature scheme for spend
authorization. There is no ECDSA, no Schnorr, no ML-DSA, and no
SLH-DSA in the authorization path.

Authorization is a zero-knowledge proof of preimage knowledge bound
to a specific state transition.

Validity-based

A block is a cryptographically proven state transition:

  pre_utxo_root, pre_nullifier_root
    ->
  post_utxo_root, post_nullifier_root

The block proof certifies that every included spend was correctly
authorized and that all state roots are correctly computed.

Proof-of-work

Block production is Sybil-resistant via proof-of-work. Specter uses
PoW because:

  - it provides external Sybil resistance without stake concentration
  - it avoids compounding the bounded-historical-trust property of
    proof pruning with a second weak-subjectivity layer from PoS
  - it aligns cleanly with a validity-chain architecture

Epoch-certified

Per-block proofs exist only within a bounded proof-retention window.
Before expiry, epoch proofs compress expired block proofs into compact
epoch validity certificates. This restores trustless historical
bootstrapping without indefinite proof storage.


/**** What Specter Is Not ****/

Specter is not a privacy coin.
Specter is not a zk-anonymity system.
Specter is not a programmable smart contract chain.
Specter is not a formally proved post-quantum theorem.
Specter is not a drop-in patch over Bitcoin.
Specter is not a claim of perfection.

Specter is a deliberate tradeoff system. The tradeoffs are named
explicitly in this document and not hidden behind branding language.

/**** Security Classes ****/

Specter explicitly separates two distinct security budgets.

Wallet Security Class (WSC-128)

Applies to: REV, all root secrets, per-output secrets, addresses,
nullifiers, and all wallet-facing hash derivations.

Primitive: BLAKE3, 256-bit output.

Target: 128-bit post-quantum preimage resistance.

Basis: Grover's algorithm provides a quadratic speedup on preimage
search. A 256-bit preimage hash requires ~2^128 Grover evaluations.
This is the accepted NIST post-quantum preimage security target for
256-bit hash functions.

Note on collision security: BLAKE3 at 256-bit output provides ~85-bit
post-quantum collision resistance under the Brassard-Hoyer-Tapp (BHT)
algorithm. This is the relevant figure for output commitment binding.
It is distinct from preimage security. 85-bit PQ collision resistance
is considered acceptable for the output commitment scheme but is not
128-bit preimage hardness. These are separate properties.

Proof Soundness Class (PSC-64)

Applies to: in-circuit hash operations (Poseidon2 over BabyBear) and
the proof-system soundness of the FRI/STARK stack.

Primitive: Poseidon2 over BabyBear; FRI-based STARKs (Plonky3).

Target: Poseidon2 at classical 128-bit security, with post-quantum
preimage security of approximately 64 bits under Grover's algorithm
applied to the 248-bit capacity.

Why PSC-64 is sufficient: the security requirement for in-circuit
hashing in Specter is proof-system soundness: no computationally
bounded adversary can forge a valid proof for a false statement with
non-negligible probability. A 64-bit quantum soundness budget exceeds
the practical attack capability assumed for the threat model. This is
not the same as the wallet-secret hardness requirement (WSC-128),
which requires 128-bit PQ preimage resistance.

PSC-64 does not achieve 128-bit post-quantum collision resistance.
Specter does not claim it does. Users who need permanent collision-
resistant historical binding should note this distinction.

PSC-64 attack cost analysis

What it means to break PSC-64: an attacker must forge a valid STARK
proof for a false statement — for example, a spend proof that claims
ownership of an output the attacker does not own. This requires
finding a Poseidon2 preimage collision or soundness exploit within
the 64-bit quantum security margin.

Grover's algorithm applied to a 64-bit security target requires 2^64
quantum oracle queries. Each query requires executing the full
Poseidon2 hash function (21 rounds over a 16-element BabyBear state)
as a coherent quantum circuit. Current estimates for the gate depth
of a single hash evaluation in a fault-tolerant quantum circuit range
from 10^3 to 10^5 logical gates, depending on implementation.

Concrete cost estimates:

  Quantum operations required:     2^64 ~ 1.8 x 10^19

  At 1 GHz quantum clock speed:    ~585 years of continuous operation
  At 1 THz quantum clock speed:    ~213 days of continuous operation

  Physical qubit cost per logical qubit (surface code, 10^-3 error
  rate): ~1,000-3,000 physical qubits per logical qubit.

  Estimated logical qubits for Grover on Poseidon2: ~500-2,000
  Estimated physical qubits: ~500,000 to 6,000,000

  For comparison: IBM's 2025 roadmap targets ~100,000 physical qubits.
  Google's Willow processor: 105 physical qubits (2024).

  Estimated cost at projected 2035 quantum computing prices:
  ~$10^9 to $10^12 per attack (order-of-magnitude estimate).

What a successful PSC-64 attack would yield: one forged proof for
one false statement. The attacker does not gain long-term advantage:
each forged spend requires a separate 2^64-operation attack. There is
no "break once, forge forever" property as with compromised signing
keys.

Why this is sufficient for proof soundness:

  - The attack cost exceeds the value of any plausible single
    transaction by many orders of magnitude.
  - Each forged proof requires a fresh attack (no key reuse analog).
  - WSC-128 (wallet secret hardness) provides the permanent security
    layer at 128-bit PQ resistance. PSC-64 protects only the
    ephemeral proof verification window.
  - If quantum hardware advances to the point where 2^64 operations
    become economically feasible, Specter's modular circuit design
    allows upgrading to a wider field without protocol redesign.

The distinction between WSC-128 and PSC-64 is not a weakness to be
hidden but a deliberate engineering tradeoff: higher in-circuit
security (wider field) would increase proving cost for every
transaction, paid by every user, to defend against an attack that
currently costs billions of dollars per instance.

Any future upgrade that raises in-circuit security would require only
a circuit change, not a protocol redesign.


/**** Core Design Thesis ****/

Specter's core thesis:

  A transparent cryptocurrency can eliminate permanent public-key
  authorization artifacts by replacing signatures with ephemeral
  zero-knowledge proofs of witness knowledge and valid state
  evolution, with historical trustlessness restored by compact
  epoch validity certificates.

Every other section in this document is a consequence of that thesis.


/**** High-Level Architecture ****/

Specter has five layers.

Layer 1: Wallet layer

Wallets hold one 256-bit REV and derive all spend authority, receive
addresses, and nullifier keys from it. Wallets maintain a rolling
witness cache: for each owned output, the wallet proactively fetches
and updates inclusion witnesses and non-membership witnesses so that
spending is ready without online latency at send time.

Layer 2: Fragment layer

Wallets do not broadcast signed transactions. They broadcast proof
fragments: state-transition proposals with zero-knowledge spend
authorization. The send flow uses a two-phase protocol (intent
broadcast followed by proof attachment) to improve UX.

Layer 3: Block production layer

Block production is separated into two economic roles:

  Block producers: select conflict-free fragment sets, determine
    canonical ordering, construct the proposed state diff, and
    recursively aggregate fragment proofs into one block proof.
    Block producers combine what earlier drafts called "assemblers"
    and "aggregators" into a single economically motivated role.

  Miners: extend the longest valid chain via proof-of-work over the
    final block header. Miners select blocks from competing block
    producers based on total payout.

Revenue is split by consensus rule: block producers receive
BLOCK_PRODUCER_FEE_SHARE percent of total fragment fees plus
BLOCK_PRODUCER_COINBASE_SHARE percent of the scheduled block reward;
miners receive the remainder of both.

Layer 4: Consensus layer

The network accepts blocks if:

  - PoW is valid
  - difficulty is correctly computed
  - the aggregated proof verifies against declared state roots
  - the state diff is consistent with declared roots
  - all consensus rules pass

Layer 5: Historical validity layer

Before per-block proofs expire, epoch compressors combine them into
epoch validity certificates. These certificates provide trustless
historical verification of all epochs within their range, with proof
size O(1) in the number of blocks per epoch.


/**** Authorization Without Public-Key Signatures ****/

In standard cryptocurrency systems, spend authorization is a relation
between a public key and a private key. A spend reveals authorization
material: a public key, a signature, or an equivalent object tied to
public-key cryptography. This leaves a permanent authorization trace
in chain history.

Specter removes that model.

In Specter, an address is not a public key or a commitment to a public
key. An address is a 32-byte BLAKE3 output derived from a secret
preimage. It contains no elliptic curve point, no lattice vector, and
no algebraic public-key structure.

Address construction

For receive index n (a 64-bit unsigned integer):

  owner_secret(n)  =  H(spend_root || LE64(n) || "specter.own")
  address(n)       =  H(owner_secret(n) || "specter.addr")

Where spend_root is derived from the wallet's REV.

Spend authorization statement

To spend an output locked to address(n), the spender proves in zero
knowledge:

  "I know owner_secret such that
   H(owner_secret || 'specter.addr') = address_in_output,
   and I am authorizing this specific state transition."

This is a proof of preimage knowledge bound to a state transition.
It is not a signature verification. The owner_secret is never
revealed. The address can be independently observed and is transparent.

What remains permanently on-chain after a spend:

  - the nullifier (32 bytes, unlinkable to owner_secret without it)
  - newly created output records
  - fee attribution
  - updated state roots

What is not permanently retained:

  - public keys
  - owner_secret
  - any public-key authorization artifact


/**** Why This Changes the Quantum Attack Surface ****/

Shor's algorithm attacks algebraic public-key structures:
elliptic-curve discrete logarithms, finite-field discrete logarithms,
and integer factorization.

Specter's permanent chain state does not expose those structures for
authorization. A post-quantum adversary examining the permanent chain
sees BLAKE3 outputs, transparent values, state commitments, and
proof-of-work headers.

There is no public authorization key to invert via Shor.

The residual quantum attack surface is:

  - Grover-style speedups on preimage search (addressed by WSC-128)
  - attacks on the in-circuit hash stack (noted as PSC-64)
  - attacks on the FRI/STARK proof system
  - implementation failures

This does not mean "quantum risk eliminated." It means Specter
removes one specific class of public authorization structures from
permanent chain history, changing the attack surface from
"Shor-vulnerable public key material" to "hash-preimage hardness
and STARK soundness."

That distinction is the point.


/**** Wallet Secret Model ****/

Each wallet holds one 256-bit master secret:

  REV ∈ {0,1}^256

From REV, the wallet derives four domain-separated roots using
BLAKE3's key derivation mode:

  id_root     =  H(REV || "specter.id")
  spend_root  =  H(REV || "specter.spend")
  null_root   =  H(REV || "specter.null")
  auth_root   =  H(REV || "specter.auth")

Root roles:

  id_root:     wallet-local scanning and account metadata only.
               Never used in spend authorization.

  spend_root:  receive-address and ownership derivation domain.

  null_root:   nullifier derivation domain.

  auth_root:   anti-replay transition-binding domain.

No root is ever published. All four are derived from REV.

Under the domain-separation model, each root is computationally
independent: knowledge of one root does not reveal another, assuming
BLAKE3 preimage hardness.


/**** Per-Output Witness Derivation ****/

For output index n:

  owner_secret(n)  =  H(spend_root || LE64(n) || "specter.own")
  address(n)       =  H(owner_secret(n) || "specter.addr")

  null_secret(n)   =  H(null_root || LE64(n) || "specter.null-key")
  auth_secret(n)   =  H(auth_root || LE64(n) || "specter.auth-key")

Each received output contains per-output uniqueness randomness:

  rho ∈ {0,1}^256

Canonical witness model

The canonical witness for spending output n is:

  w_n = (
    spend_root,
    null_root,
    auth_root,
    n,
    rho,
    inclusion_witness,       -- MMR path proving output_cm is in UTXO set
    nonmembership_witness    -- IMT path proving nullifier is not yet spent
  )

The proving relation derives inside the circuit:

  owner_secret(n)  from  (spend_root, n)
  null_secret(n)   from  (null_root, n)
  auth_secret(n)   from  (auth_root, n)
  address(n)       from  owner_secret(n)
  output_cm        from  (address, value, rho)
  nullifier        from  (null_secret(n), output_cm)
  auth_tag         from  (auth_secret(n), transition_hash)

This derivation structure is critical. All three secret branches are
derived from the same REV roots and the same index n. An adversary
who knows one branch cannot forge the others. Partial-witness attacks
are closed at the protocol level.

Security consequence

To forge a valid spend, an adversary must either:

  - know a valid witness consistent with the output and the transition,
  - break the proving relation soundness, or
  - break the relevant hash preimage assumptions.

This is the exact intended security boundary.


/**** Output Model ****/

A Specter output is a triple:

  output = (address, value, rho)

  address   32 bytes   BLAKE3 output, transparent
  value      8 bytes   unsigned 64-bit integer, base units, transparent
  rho       32 bytes   per-output uniqueness randomness

Output commitment

  output_cm = H(
    address         ||
    LE64(value)     ||
    rho             ||
    "specter.output"
  )

This commitment enters the UTXO accumulator.

Permanent output record

Each output is permanently stored as:

  output_cm   32 bytes
  address     32 bytes
  value        8 bytes
  rho         32 bytes
  ────────────────────
  Total:      104 bytes per output record

Output records are transparent and sufficient for: supply auditing,
wallet scanning by derived address, accumulator reconstruction, and
block explorer indexing.

Same-block spend rule

An output created at block height h cannot be consumed until height
h+1. This is a hard consensus rule. It eliminates intra-block
dependency ordering from the aggregation circuit semantics and
simplifies canonical state-diff application.


/**** Nullifier Construction ****/

Each spend consumes exactly one prior output and produces exactly one
nullifier for that output.

Nullifier derivation (as computed inside the proving circuit):

  null_secret(n) = H(null_root || LE64(n) || "specter.null-key")

  nullifier = H(
    null_secret(n) ||
    output_cm      ||
    "specter.nullifier"
  )

This two-stage construction provides three properties:

  Uniqueness
    The nullifier is unique per (null_secret, output) pair because
    output_cm binds rho and the full output contents.

  Spend unlinkability
    An observer who sees a nullifier cannot map it to its address
    without knowing null_root. The address and nullifier derivations
    use different root branches.

  Double-spend prevention
    Each output generates exactly one valid nullifier under the
    witness relation. Reusing an output would require either forging
    a second valid nullifier or spending an already-spent nullifier,
    both of which violate the consensus rules.

Nullifiers are the permanent spentness memory of the chain.

Specter does not eliminate permanent historical state. It relocates
permanent spend evidence from signatures and public keys to nullifiers.
A nullifier is 32 bytes. An ML-DSA-65 signature is 3,309 bytes.
That is a 100x reduction in permanent per-spend storage.


/**** Authorization Binding ****/

Spend authorization must be bound to the specific state transition
being proposed, not merely to output ownership. Otherwise, a valid
witness could be replayed against a different transition.

Transition hash

  transition_hash = H(
    "specter.transition"     ||
    anchor_root              ||
    consumed_nullifiers[]    ||
    created_outputs[].output_cm ||
    LE64(fee)
  )

Authorization tag (per consumed output, derived inside circuit):

  auth_secret(n) = H(auth_root || LE64(n) || "specter.auth-key")

  auth_tag = H(
    auth_secret(n)  ||
    transition_hash ||
    "specter.auth"
  )

The auth_tag is not permanently retained in canonical chain data.
It exists only inside the proving relation during the proof window.

This binding prevents:

  - replay against different output sets
  - replay against different fee amounts
  - replay against different state anchors
  - replay outside the anchor window

Authorization is not "I know the owner secret." It is:

  "I know the correct witness for these consumed outputs
   and I am explicitly authorizing this specific transition."


/**** Anchor-Window Binding ****/

Binding a proof fragment to one exact pre-state root is too brittle
for a practical mempool. If the chain advances by even one block,
every pending fragment is immediately stale. Under contention this
forces wallets to reprove continuously, which is both expensive and
hostile to UX.

Specter uses anchor-window binding instead.

Each fragment references:

  anchor_height   the block height the fragment was built against
  anchor_root     the canonical state root at that height

A fragment is valid for inclusion in a block at height h if:

  anchor_height <= h <= anchor_height + ANCHOR_WINDOW

Consensus parameters:

  ANCHOR_WINDOW      = 8 blocks   (~16 minutes at 120-second blocks)
  INTENT_EXPIRY_WINDOW = 4 blocks  (~8 minutes at 120-second blocks)

ANCHOR_WINDOW governs fragment validity for block inclusion.
INTENT_EXPIRY_WINDOW governs how long a SpendIntent without an
attached SpendFragment is retained in the mempool.

These are distinct parameters with distinct purposes. Conflating
them would prevent independent tuning: INTENT_EXPIRY_WINDOW should
be short to limit spam-slot occupation; ANCHOR_WINDOW should be
long enough to allow proof completion and relay without excessive
reproving. A 2:1 ratio (4 vs 8 blocks) is the v1 choice.

At 120-second block times: approximately 16 minutes of fragment
validity before reproving is required.

In-circuit validity relation

The proof does not claim:

  "this transition is valid only against the immediate tip root"

It claims:

  "this transition is valid against a state descended from anchor_root,
   provided the consumed outputs are still unspent and the derived
   nullifiers are still absent at the anchor state."

Block-producer obligation

A block producer MUST verify, outside the fragment proof, that at
inclusion time:

  - all consumed outputs are still present in the UTXO accumulator
  - all nullifiers are still absent from the nullifier set
  - the fragment's anchor window has not expired
  - WitnessBundles are published for all outputs included in the block
    (see Witness Bundle Protocol)

This moves some conflict-resolution burden into block production logic,
which is acceptable and preferable to forcing wallets to reprove
on every chain tip movement.


/**** Two-Phase Spend Path ****/

The standard spend proof takes 1–5 seconds to generate on CPU. If
wallets must complete a full proof before any network-visible action,
the UX is equivalent to:

  "nothing happens for 4 seconds, then the send appears"

This is unacceptable for practical use. Specter defines a two-phase
spend protocol.

Phase 1: Intent Broadcast

The wallet immediately broadcasts a SpendIntent before proof
generation completes:

  SpendIntent:
    intent_id             = H("specter.intent" || anchor_height ||
                              anchor_root || consumed_output_ids[] ||
                              created_output_cms[] || LE64(fee))
    anchor_height         8 bytes
    anchor_root          32 bytes
    consumed_output_ids[]  32 bytes each
    created_output_cms[]   32 bytes each
    fee                   8 bytes
    intent_commitment    32 bytes   H(above fields)

The SpendIntent is not a proof. It is a network-visible claim of
intended spend. This allows:

  - mempools to reserve conflict slots
  - block producers to begin preliminary block layout
  - wallets to show "pending" state to users immediately

Intent anti-spam rules

SpendIntents are cheap to produce (no proof required) and could be
used to flood mempools and lock output IDs for up to INTENT_EXPIRY
blocks without any real spend following. The following rules MUST be
enforced by all mempool implementations:

  1. One active intent per output ID.
     A mempool MUST NOT accept a second intent for an output ID that
     is already reserved by an active intent. If the first intent
     expires or is superseded by a valid fragment, the lock is
     released.

  2. Intent rate limit per peer connection.
     Nodes SHOULD enforce a per-peer rate limit of:
       MAX_INTENTS_PER_PEER_PER_BLOCK = 10
     Intent floods from a single peer are a relay-layer attack.
     Peers exceeding this rate SHOULD be disconnected and scored
     negatively in peer scoring.

  3. Intent fee plausibility check.
     A node SHOULD reject intents with declared fee below the local
     minimum fee policy for the declared weight class. An intent
     with a fee that would never qualify for block inclusion is
     either an error or spam.

  4. Intent expiry enforcement.
     Nodes MUST evict all intents that have reached
     (anchor_height + INTENT_EXPIRY_WINDOW) without a matching
     SpendFragment. Evicted intents release their output locks.
     Nodes MUST NOT retain expired intents in memory.

  5. Intent-to-fragment matching.
     When a SpendFragment arrives, nodes MUST verify that the
     fragment's consumed_nullifiers[] and created_output_cms[]
     exactly match the corresponding fields in the SpendIntent
     referenced by intent_id. Mismatched fragments are rejected.
     This prevents a fragment from being attached to the wrong
     intent.

  6. Duplicate intent rejection.
     Intents are deduplicated by intent_id. A node that has already
     seen and accepted an intent with a given intent_id MUST silently
     drop duplicate relays.

Phase 2: Proof Attachment

The wallet broadcasts a SpendFragment after proof completion:

  SpendFragment:
    intent_id            32 bytes
    public_inputs_blob   variable
    spend_proof          variable (~30-100 KB)

The SpendFragment must arrive before the intent expires. Intent
expiry is governed by:

  INTENT_EXPIRY_WINDOW = 4 blocks   (~8 minutes at 120-second blocks)

This is intentionally shorter than ANCHOR_WINDOW (8 blocks). A
wallet that cannot complete proof generation within 4 blocks can
re-broadcast a new SpendIntent at a fresh anchor before its proof
completes. This reduces the spam-slot occupation window by 2x
compared to conflating intent expiry with the full anchor window.

If proof attachment does not arrive within the window, the intent
is dropped and the output locks are released for re-announcement.

Mempools MUST accept intents but MUST NOT include outputs in
assembled blocks until a valid SpendFragment is attached.

UX consequence

From the user's perspective:

  click "send" → "pending" state appears immediately
  1-5 seconds later → "in mempool" state appears
  next block → "confirmed"

This is substantially better than:

  "spinner" for 4 seconds → "in mempool" appears


/**** Witness Bundle Protocol ****/

To generate a fragment proof, a wallet needs:

  - an MMR inclusion proof for each consumed output
  - an IMT non-membership proof for each derived nullifier
  - a current anchor root and height

These must come from a full node. The naive design — wallet makes
a one-off request to one trusted node — is fragile and censorship-
susceptible.

Specter defines a witness bundle protocol.

Witness bundle

  WitnessBundle:
    bundle_id             = H("specter.witness" || output_id ||
                              anchor_height || anchor_root)
    output_id            32 bytes
    anchor_height         8 bytes
    anchor_root          32 bytes
    inclusion_witness    ~27 x 32 bytes   MMR path
    nonmembership_witness ~32 x 32 bytes  IMT path
    bundle_expiry         8 bytes   (anchor_height + ANCHOR_WINDOW)

Witness bundles are self-verifying: a recipient can verify the
inclusion and non-membership proofs independently against the claimed
anchor_root, without trusting the provider.

Witness serving is not ancillary infrastructure; it is part of spend
availability. A wallet that cannot obtain a valid WitnessBundle cannot
spend its outputs, regardless of how correct the cryptography is. The
availability of witness serving is as important to Specter's liveness
as block production itself.

Distribution

Witness bundles are served through two complementary mechanisms:

  1. Block producer obligation (primary):
     Block producers MUST publish WitnessBundles for all new outputs
     created in each block they produce. This includes all outputs in
     the state_diff's new_outputs[] list. The block producer already
     maintains the full state (MMR, IMT) required to construct these
     bundles; the marginal cost of serving them is small relative to
     the proving cost already borne.

     This obligation piggybacks witness serving on an already-
     incentivized role. Block producers are compensated via the
     consensus fee split and coinbase subsidy, and have no economic
     reason to withhold
     witnesses for outputs they included.

  2. Full node p2p obligation (secondary):
     Every consensus full node MUST respond to
     GetWitnessBundle(output_id, anchor_height) requests from peers.
     This is a p2p protocol obligation. Full nodes possess the complete
     MMR and IMT state required to construct any WitnessBundle on
     demand. Non-responsive nodes are deprioritized in peer scoring.

     This obligation is not consensus-enforced (a node cannot be
     slashed for non-response) but is part of the expected behavior
     of a conforming full node, similar to how Bitcoin full nodes
     are expected to serve block data to peers.

Wallets do not request witnesses on-demand at send time. Instead,
each wallet maintains a rolling witness cache:

  - on receiving an output, immediately request its inclusion witness
    (typically from the block producer that included the output)
  - periodically refresh witnesses as the accumulator advances
  - maintain pre-computed spend-ready state for all owned outputs

Send-time flow:

  wallet selects output -> witness is already cached
  -> proof generation can begin immediately without network round-trip
  -> anchor rebasing updates the attachment header if needed

Multi-source verification:

  Wallets SHOULD request each WitnessBundle from at least two
  independent sources (e.g., the block producer and a full node peer)
  and verify both against the anchor_root. Divergent bundles indicate
  either a provider fault or a witness relay eclipse attempt. The
  wallet MUST use the bundle that verifies correctly and SHOULD log
  the discrepancy.

Witness availability fault

If a wallet cannot obtain a WitnessBundle from any source after
exhausting its peer list:

  - The wallet MUST NOT attempt to generate a proof without a
    verified WitnessBundle. Proofs generated without valid witnesses
    will fail verification.
  - The wallet SHOULD notify the user that spend availability is
    degraded and suggest trying again after the network recovers.
  - The wallet SHOULD try at minimum 3 distinct sources (block
    producer + 2 full node peers) before declaring unavailability.

Anchor rebasing

If the chain advances between witness fetch and proof submission, the
wallet may rebase the proof's anchor header without reproving the
core spend authorization.

The spend circuit is split into two logically separable components:

  Core circuit (ownership proof):
    Proves knowledge of spend_root, null_root, auth_root, and index n
    such that the derived owner_secret, nullifier, and auth_tag are
    all internally consistent with the consumed output and the
    transition_hash. The core circuit does not directly reference
    anchor_height or anchor_root. It proves ownership, not freshness.

  Attachment circuit (freshness proof):
    Proves that:
      - the MMR inclusion path for the consumed output is valid at
        the claimed anchor_root
      - the IMT non-membership path for the derived nullifier is
        valid at the claimed anchor_root
      - anchor_height is within the valid anchor window for the
        block being assembled

The attachment circuit references anchor_height, anchor_root, and the
path witnesses from the WitnessBundle. It is the separable component.

Rebasing procedure:

  1. Wallet fetches a fresh WitnessBundle at new anchor_height', anchor_root'.
  2. Wallet runs only the attachment circuit with the new path witnesses
     and new anchor_root', producing a fresh attachment proof.
  3. Wallet replaces the old attachment proof in the SpendFragment
     with the new one. The core ownership proof is unchanged.
  4. Wallet recomputes transition_hash if the anchor_height changed,
     since transition_hash includes anchor_root. If transition_hash
     changes, the auth_tag changes and the core circuit MUST be
     re-run for authorization binding only.

Rebasing constraint: transition_hash binds to anchor_root. If
anchor_root changes in a rebase, the auth_tag derived from
auth_secret and transition_hash changes, requiring a partial
re-run of the core circuit's authorization binding portion. This
is significantly cheaper than full re-proving because the ownership
and nullifier portions of the core circuit can be pre-committed and
reused. Exact savings depend on the AIR implementation; implementers
SHOULD structure the core circuit to allow partial re-execution of
the auth_tag derivation without re-running the full witness relation.

This reduces reproving events from "any chain movement" to "more than
ANCHOR_WINDOW blocks passed," an approximate ANCHOR_WINDOW-fold
improvement in normal network operation.


/**** Fragment Format ****/

A proof fragment is the unit of mempool propagation.

  Fragment:
    version               4 bytes
    anchor_height         8 bytes
    anchor_root          32 bytes
    consumed_nullifiers[] 32 bytes each
    created_outputs[]    104 bytes each
    fee                   8 bytes
    fragment_id          32 bytes
    public_inputs_blob   variable
    spend_proof          ~30-100 KB (ephemeral)
    weight_class          1 byte  (W1, W2, W3; see Fee Market)

Fragment ID:

  fragment_id = H(
    "specter.fragment"      ||
    anchor_height           ||
    anchor_root             ||
    consumed_nullifiers[]   ||
    created_outputs[].output_cm ||
    LE64(fee)
  )

Fragments are deduplicated by fragment_id.

Fragment validity (mempool admission checks):

  - spend_proof verifies against public inputs
  - anchor window is live: h - anchor_height <= ANCHOR_WINDOW
  - fee >= minimum fee for declared weight_class
  - consumed nullifiers are not already known-spent in mempool
  - referenced outputs exist and are unspent
  - created outputs are well-formed
  - no intra-fragment nullifier duplication

Fragments that fail any check are rejected before block assembly.

Weight classes (v1):

  W1:  1 consumed output, 1-2 created outputs (standard send)
  W2:  2 consumed outputs, 1-3 created outputs (consolidation)
  W3:  3-4 consumed outputs, 1-4 created outputs (batch merge)

Inputs or outputs beyond W3 limits are not supported in v1.

This constraint minimizes the proving surface and AIR complexity.
A fixed-layout circuit is easier to audit and harder to misspecify
than an arbitrary-input circuit. Larger fan-in is a future extension.

Fixed-width circuit layout and padding

Each weight class corresponds to exactly one circuit with a fixed
number of input slots and output slots:

  W1 circuit:  1 input slot, 2 output slots
  W2 circuit:  2 input slots, 3 output slots
  W3 circuit:  4 input slots, 4 output slots

A fragment using fewer than the maximum slots for its class MUST pad
unused slots with canonical dummy values. Dummy values are:

  Dummy consumed output:
    owner_secret_n  = H("specter.dummy.spend" || LE64(slot_index))
    null_secret_n   = H("specter.dummy.null"  || LE64(slot_index))
    auth_secret_n   = H("specter.dummy.auth"  || LE64(slot_index))
    address         = H(dummy_owner_secret_n || "specter.addr")
    value           = 0
    rho             = H("specter.dummy.rho"   || LE64(slot_index))

  Dummy created output:
    address         = H("specter.dummy.out"   || LE64(slot_index))
    value           = 0
    rho             = H("specter.dummy.rho2"  || LE64(slot_index))

The circuit enforces: dummy slots do not alter value conservation
(dummy value is 0, dummy fee contribution is 0), and dummy nullifiers
are not included in the fragment's consumed_nullifiers[] list.

This ensures all proofs of the same weight class have identical
circuit size, identical trace dimensions, and identical proving and
verification time, regardless of actual input/output count. This is
required for correct binary tree aggregation.

Rationale: a variable-layout circuit creates variable proof sizes
and variable recursion tree depths, which complicates aggregation
and makes AIR auditing substantially harder. Fixed layout eliminates
both problems.

A W1 fragment with 1 input and 1 output uses the W1 circuit with
one dummy created output slot. A W2 fragment with 2 inputs and
1 output uses the W2 circuit with two dummy created output slots.


/**** Fragment Relay Bandwidth ****/

Fragments carry proof blobs of approximately 30–100 KB each. At
target throughput (500 fragments per block, one block every 120
seconds), the per-block fragment propagation load is:

  Minimum: 500 x 30 KB  =  15 MB per block  (~125 KB/s sustained)
  Maximum: 500 x 100 KB = 50 MB per block   (~417 KB/s sustained)

This is the ephemeral relay bandwidth requirement for nodes that
participate in the mempool. It is not permanent storage — proofs are
pruned after PROOF_RETENTION_WINDOW. However, it is a real bandwidth
cost that affects who can operate a full node with mempool
participation.

Node relay requirements:

  Consensus full node (non-mining):
    Inbound fragment relay: ~125–417 KB/s sustained at full blocks
    Outbound: serves witness bundles and block data; no proof relay
              obligation unless the node opts into proof archiving

  Block-producing node:
    Must receive all valid fragments within the block time window.
    Must relay the aggregated block proof (~50-200 KB) on block
    production. Must serve witness bundles for included outputs.
    Inbound: ~125-417 KB/s + standard block relay overhead
    Outbound: block propagation (~200 KB proof + state_diff) +
              witness bundles for included outputs

  Light client:
    No fragment relay requirement. Downloads block headers and
    EpochCertificates only.

Mempool relay policy:

  Nodes SHOULD relay only fragments that pass full mempool admission
  checks (see Fragment validity above) before forwarding. Relaying
  unverified proof blobs enables bandwidth amplification attacks.
  A proof that fails verification on arrival was wasted bandwidth
  on all hops. The relay rule is:

    verify proof -> pass admission checks -> relay

  This is more expensive per-hop than Bitcoin's signature check relay
  policy (~0.1 ms for ECDSA vs ~5-20 ms for a STARK verification),
  but it is O(1) and independent of proof complexity.

Fragment size bound:

  The maximum fragment proof blob size is protocol-bounded at 128 KB.
  Proofs exceeding 128 KB are rejected at mempool admission. This
  provides an upper bound for bandwidth planning. Actual proof sizes
  under current Plonky3/BabyBear are expected to remain within
  50-100 KB for W1/W2/W3 circuits.


/**** What the Spend Proof Proves ****/

The per-fragment STARK proof proves the following statements jointly.

For each consumed output:

  Ownership
    The prover knows spend_root and index n such that:
      owner_secret(n) = H(spend_root || LE64(n) || "specter.own")
      address = H(owner_secret(n) || "specter.addr")
    and this address matches the consumed output record.

  Nullifier correctness
    The prover knows null_root and the same index n such that:
      null_secret(n) = H(null_root || LE64(n) || "specter.null-key")
    and the derived nullifier matches the claimed consumed nullifier.

  Authorization binding
    The prover knows auth_root and the same index n such that:
      auth_secret(n) = H(auth_root || LE64(n) || "specter.auth-key")
    and auth_tag binds correctly to transition_hash.

  Output commitment consistency
    output_cm = H(address || LE64(value) || rho || "specter.output")
    matches the claimed output record.

  UTXO accumulator inclusion
    The consumed output exists in the MMR at or after anchor_root,
    via the provided MMR inclusion path.

  Nullifier set non-membership
    The derived nullifier is absent from the nullifier IMT at
    anchor_root, via the provided IMT non-membership path.

Across the fragment:

  Value conservation
    sum(consumed_values) = sum(created_values) + fee

  Output well-formedness
    Each created output record and commitment is internally consistent.

  Transition binding
    Fragment authorization binds to transition_hash and anchor
    window semantics.

  No intra-fragment duplication
    No nullifier or output appears twice within the same fragment.

  Weight class compliance
    Fragment input/output counts are within the declared weight class.

This is the protocol-level replacement for "signature validity."


/**** Block Production Roles ****/

Specter separates block production into two economic roles: block
producers and miners. Earlier drafts described three roles (assembler,
aggregator, miner). In practice, the assembler and aggregator have no
independent revenue stream and would either vertically integrate with
miners or centralize into unpaid intermediaries. This is not
theoretical: Ethereum PBS shows 2-3 builders produce >90% of blocks;
Mina's SNARK workers are driven to marginal-cost pricing; every major
ZK-rollup operates a centralized prover funded internally.

Specter accepts that proof aggregation will exhibit hardware-driven
concentration and addresses it by: (a) making the block producer a
single, explicitly compensated role; and (b) keeping the miner role
permissionless, lightweight, and decoupled from proving hardware.

Block Producers

A block producer observes the mempool, selects a conflict-free set of
valid fragments, establishes canonical ordering, constructs the
proposed state diff, and produces the aggregated block proof by
recursively combining fragment proofs.

  BlockCandidate:
    candidate_id          = H("specter.candidate" || height ||
                              ordered_fragment_ids[] || proposed_state_diff)
    height                8 bytes
    ordered_fragment_ids[] 32 bytes each
    proposed_state_diff   (new_nullifiers[], new_outputs[], fee_kernels[])
    pre_state_root        32 bytes
    expected_post_root    32 bytes
    producer_address     32 bytes   block producer's receive address
    aggregated_proof     ~50-200 KB
    proof_commitment     32 bytes   H("specter.block-proof" || aggregated_proof)

Fragment ordering (canonical):

  1. Primary: descending fee-rate (fee / weight_class_cost)
  2. Tie-break: ascending fragment_id (deterministic)

Proof aggregation uses binary tree recursion with canonical padding.

Tree construction for N fragments:

  1. Compute K = smallest power of 2 >= N.
  2. Pad the fragment list to length K by appending (K - N) canonical
     trivial proofs. A trivial proof is the proof of an empty valid
     transition: zero consumed outputs, zero created outputs, zero fee,
     with a well-formed anchor and correct dummy witness. Trivial proofs
     are pre-computed once per block and reused.
  3. Arrange the K padded proofs as leaves of a complete binary tree
     of depth ceil(log2(K)) = log2(K).
  4. Recursively aggregate pairwise from the leaves up:
       Level 0:         K leaf proofs (N real + K-N trivial)
       Level 1:         K/2 pairwise aggregations
       ...
       Level log2(K):   1 block proof

For N = 500 fragments: K = 512, depth = 9 levels.
For N = 1 fragment:    K = 1, depth = 0 (block proof = fragment proof).

Canonical leaf ordering: fragments in their block-producer-assigned
order (descending fee_score, tie-broken by ascending fragment_id),
followed by trivial proofs. This makes the tree structure
deterministic given the ordered_fragment_ids[] in the BlockCandidate.

The trivial proof circuit is the W1 circuit with all dummy inputs and
zero fee. Its public inputs are:
  consumed_nullifiers[] = []
  created_outputs[]     = []
  fee                   = 0
  anchor_height         = block height
  anchor_root           = utxo_root_pre

Trivial proofs contribute nothing to state effects and are not
included in the state_diff. They exist only to complete the binary
tree structure.

Block producers require GPU-class proving hardware for competitive
aggregation within the block time window.

Miners

A miner takes a finalized BlockCandidate (header + state_diff +
aggregated proof) and performs proof-of-work over the header. The
miner does not need to perform aggregation: they receive completed
BlockCandidates from block producers and select the best one.

Miner selection criteria: miners SHOULD select the BlockCandidate
that maximizes their payout. Competing block producers deliver
candidates with varying total fees; the miner's share is determined
by the consensus fee split rule (see Fee Distribution below).

This two-role separation keeps each role focused:

  Block producer: GPU-required, competitive, explicitly compensated
  Miner: PoW hardware, independent of proving hardware, permissionless

The roles are separable but not required to be separate. A single
operator MAY perform both roles simultaneously — setting both
coinbase_address and producer_address to the same wallet. The protocol
is indifferent. What matters is that the system works correctly in
both configurations: when the roles are held by different parties
(the competitive case) and when one operator performs both (the
vertically integrated case). The fee split is enforced by consensus
regardless of whether the addresses belong to the same entity.


/**** Fee Distribution ****/

In Specter, the most expensive computational role in block production
is proof aggregation, not PoW hashing. A system that pays only the
miner while the block producer bears the highest cost creates an
unsustainable asymmetry. Every comparable system that leaves the
proving role uncompensated converges to one of two failure modes:
vertical integration (miner becomes prover, defeating role separation)
or centralization into 2-3 dominant operators subsidized off-protocol.

Specter addresses this with a consensus-enforced fee split.

Design principle: The protocol enforces a builder subsidy at the
consensus layer to ensure long-term viability of independent proof
generation and prevent the emergence of off-chain subsidy markets.
The split is not optional, not configurable per-block, and not
subject to miner veto. If the coinbase outputs do not match the
prescribed split, the block is invalid. This is a deliberate protocol
choice: proving infrastructure must be compensated on-chain, by the
protocol itself, to avoid centralizing pressure that off-protocol
arrangements inevitably create.

When builder == miner (same operator controls both addresses), the
operator receives 100% of all revenue. The protocol does not
distinguish this case — the split is still computed and enforced,
but both outputs flow to the same entity. This is the expected
configuration for many operators and is explicitly supported.

Fee and coinbase split rule (consensus-enforced):

  BLOCK_PRODUCER_FEE_SHARE     = 30   (30% of fees to block producer)
  BLOCK_PRODUCER_COINBASE_SHARE = 10   (10% of coinbase to block producer)

  total_fragment_fees = sum(fee_amount for all fragments in block)
  scheduled_reward    = scheduled_reward(height)

  producer_fee     = (total_fragment_fees * BLOCK_PRODUCER_FEE_SHARE) / 100
  miner_fee        = total_fragment_fees - producer_fee

  producer_subsidy = (scheduled_reward * BLOCK_PRODUCER_COINBASE_SHARE) / 100
  miner_reward     = scheduled_reward - producer_subsidy

The miner receives miner_reward + miner_fee. The block producer
receives producer_subsidy + producer_fee. The producer subsidy
guarantees a minimum income from every block regardless of fee
volume, solving the bootstrap problem for independent block producers.

Coinbase construction under fee split:

Each block produces two coinbase outputs:

  MinerCoinbase:
    address   = coinbase_address   (miner's receive address, from header)
    value     = miner_reward + miner_fee
    rho       = H("specter.coinbase.miner" || height || prev_hash)

  ProducerCoinbase:
    address   = producer_address   (block producer's receive address)
    value     = producer_subsidy + producer_fee
    rho       = H("specter.coinbase.producer" || height || prev_hash)

Both coinbase outputs use the standard output model and enter the UTXO
MMR as standard output records. Canonical ordering in new_outputs[]:
miner coinbase first, producer coinbase second, then all other outputs.

Both coinbase outputs are subject to COINBASE_MATURITY (100 blocks)
and EPOCH_DEBT_PENALTY for uncertified epochs, identically to the
single-coinbase model.

The ProducerCoinbase is ALWAYS present because producer_subsidy > 0
whenever the scheduled block reward is non-zero. Even in empty blocks
(zero fees), the block producer receives the coinbase subsidy as
compensation for the aggregation proof.

Consensus validation of fee split:

Block validation step 8 (monetary correctness) is extended:

  8. Monetary correctness
     total_fragment_fees = sum(fee_amounts in state_diff)
     reward = scheduled_reward(height)

     producer_fee     = (total_fragment_fees * BLOCK_PRODUCER_FEE_SHARE) / 100
     miner_fee        = total_fragment_fees - producer_fee
     producer_subsidy = (reward * BLOCK_PRODUCER_COINBASE_SHARE) / 100
     miner_reward     = reward - producer_subsidy

     miner_coinbase_value    = miner_reward + miner_fee
     producer_coinbase_value = producer_subsidy + producer_fee

     Verify both coinbase outputs match expected values.
     Verify ProducerCoinbase is always present when reward > 0.

All integer divisions use floor division. Any remainder (from
rounding) goes to the miner. This is deterministic and consensus-safe.

Why 30/70 fees + 10% coinbase subsidy: The proving cost for recursive
STARK aggregation of ~500 fragments on GPU hardware is estimated at
$0.01-0.10 per block at current hardware prices (2025-2026 benchmarks
show per-proof costs of ~$0.02 for modern STARK provers, with per-
transaction costs as low as $0.004 at scale). This cost is low and
falling — Circle STARKs, hardware acceleration, and prover
competition continue to drive it down.

The 30% fee share alone covers proving cost at any reasonable fee
level. The coinbase subsidy (BLOCK_PRODUCER_COINBASE_SHARE = 10%)
serves a different purpose: it guarantees a minimum revenue floor
that justifies the block producer's infrastructure investment (GPU
hardware, bandwidth, mempool monitoring) independent of fee volume.
Without this floor, during low-fee phases — early network, periods
of low activity — the rational response is to shut down proving
infrastructure, reducing block production competition to a single
vertically integrated miner-builder. The subsidy prevents this
centralization pressure by ensuring that operating a block producer
node is always economically viable, even when fees are negligible.

As the network matures and fees grow, the subsidy becomes a smaller
fraction of total producer revenue, and the fee share dominates.

These parameters (BLOCK_PRODUCER_FEE_SHARE = 30,
BLOCK_PRODUCER_COINBASE_SHARE = 10) are v1 values calibrated to
current hardware economics. Future protocol upgrades may adjust
either parameter based on observed proving costs, fee levels, and
network conditions.


/**** Block Production Economics ****/

Cost structure analysis (2025-2026 estimates, based on real benchmarks):

  Proving cost (recursive STARK aggregation, ~500 fragments):
    GPU class       : A100 / H100 (or consumer RTX 4090/5090)
    Per-proof cost  : ~$0.02 (Succinct SP1 Hypercube benchmark)
    Per-tx cost     : ~$0.004 (zkSync Era at scale)
    Total per block : $0.01 - $0.10 (baseline ~$0.05)
    Trend           : falling rapidly (Circle STARKs, S-two prover,
                      hardware acceleration, prover competition)

  Mining cost (PoW, at equilibrium):
    Hardware + electricity per block: $80 - $120 (baseline ~$100)

  Cost ratio:
    proving / total  < 0.1%  (proving cost is negligible vs. mining)

Revenue analysis under the dual-split model:

  Assume: block_reward = $100, total_fees = $20

  Producer revenue:
    coinbase subsidy = 10% * $100 = $10.00
    fee share        = 30% * $20  = $6.00
    total            = $16.00

  Miner revenue:
    coinbase share   = 90% * $100 = $90.00
    fee share        = 70% * $20  = $14.00
    total            = $104.00

  Producer margin at baseline proving cost ($0.05):
    $16.00 - $0.05 = +$15.95  (highly profitable)

  Producer margin at zero fees (early network):
    coinbase subsidy only = $10.00
    $10.00 - $0.05 = +$9.95  (profitable from subsidy alone)

This model reveals three important dynamics:

  1. Proving cost is NOT the bottleneck. At $0.01-$0.10 per block,
     the computational cost of proof generation is trivial. The real
     cost of running a block producer is infrastructure: GPU hardware
     purchase/lease ($24K-$120K for a competitive cluster), bandwidth
     for mempool monitoring, and operational overhead. The coinbase
     subsidy compensates for this infrastructure investment, not for
     per-block marginal proving cost.

  2. The subsidy justifies infrastructure commitment. A block producer
     who invests $50K in GPU hardware needs predictable revenue to
     justify that capital expenditure. The coinbase subsidy ($10 per
     block at $100 reward) provides ~$1,440/day at 144 blocks/day,
     covering hardware costs and ensuring multiple independent
     producers can compete rather than a single integrated operator.

  3. Vertical integration: A miner-builder who performs both roles
     captures 100% of both coinbase and fees. This is always more
     profitable than separation. The split parameters ensure that
     separation is VIABLE, not that it is optimal. The market will
     contain a mix of integrated operators and specialized builders,
     with the ratio determined by hardware economics.

Producer break-even analysis:

  Per-block proving cost is negligible (~$0.05). The real question
  is whether producer revenue covers infrastructure amortization.

  At 144 blocks/day, $100 reward, $20 avg fees:
    daily producer revenue = 144 * $16 = $2,304
    annual producer revenue = ~$841K

  At 144 blocks/day, $100 reward, $0 fees (early network):
    daily producer revenue = 144 * $10 = $1,440
    annual producer revenue = ~$526K

  A competitive GPU cluster ($50K-$120K) is paid off within months
  even in the zero-fee scenario. The subsidy makes independent block
  production viable from day one.


/**** Block Production Market Structure ****/

Expected market dynamics:

Block production (fragment selection + proof aggregation) will exhibit
hardware-driven concentration. GPU economics, bandwidth requirements,
and latency advantages create barriers to entry that favor well-
equipped operators. This is an accepted structural property, not a
defect to be designed away.

The key structural insight: concentration at the block producer layer
is non-critical to security because the miner retains permissionless
selection power. A miner can choose between competing block producers
based on payout, latency, or any other criterion. The consensus rule
only requires that the block proof verifies and the fee split is
correct. Who produced the proof is irrelevant to consensus.

Competition mechanisms:

  1. Multiple block producers per height.
     Any block producer can submit BlockCandidates to any miner.
     Miners are not bound to a single block producer.

  2. Fee competition.
     Block producers compete on the quality of their fragment
     selection (higher total fees = higher miner payout = more
     miners select their candidate).

  3. Latency competition.
     Block producers who aggregate faster deliver candidates to
     miners earlier, giving miners more PoW time on that candidate.

  4. Deterministic fragment ordering.
     Fragment ordering is canonical (descending fee_score, tie-broken
     by ascending fragment_id). A block producer cannot arbitrarily
     reorder fragments to extract value beyond the fee-score ordering.
     This limits the MEV surface compared to systems with
     unconstrained transaction ordering.

  5. Open proving stack.
     Plonky3 is open source. GPU acceleration via ICICLE is open
     source. No proprietary hardware or software is required to
     operate as a block producer.

Censorship considerations:

A block producer who excludes specific fragments from their candidate
reduces their total fee revenue and makes their candidate less
attractive to miners. A censoring block producer is outcompeted by a
non-censoring one with higher total fees. This does not eliminate
censorship (a dominant block producer could still censor at cost) but
creates an economic penalty for it.

If censorship becomes systemic, miners can switch to alternative block
producers or produce their own blocks (at the cost of operating
proving hardware). The PoW layer remains the ultimate censorship
backstop.


/**** UTXO Accumulator ****/

Unspent outputs are tracked in a Merkle Mountain Range (MMR),
following the Utreexo-style accumulator model (Dryja, ePrint 2019/611).

An MMR provides:

  - append-only insertions
  - O(log N) inclusion proofs
  - efficient root commitments
  - no in-place updates required

The MMR root appears in every block header as utxo_root_pre and
utxo_root_post, committing to the unspent-output set before and after
each block.

Inclusion proof size

For a UTXO set of N entries, an MMR inclusion proof consists of
approximately log_2(N) hash values of 32 bytes each.

For N = 100,000,000 entries: ~27 hashes x 32 bytes = ~864 bytes.

This is the per-spend witness size. With the rolling wallet witness
cache, this fetch occurs at output receipt time, not at send time.

The MMR inclusion witness is a consensus-critical input to fragment
proof verification.


/**** Nullifier Set Structure ****/

Nullifiers are stored in an Indexed Merkle Tree (IMT). The IMT
structure was described in "Transparency Dictionaries with Succinct
Proofs of Correct Operation" (ePrint 2021/1263, page 6) and was
deployed for nullifier tracking by Aztec Protocol.

The IMT is a sorted linked-list structure that enables predecessor-
style non-membership proofs.

Why IMT over sparse Merkle tree

Proving non-membership against a sparse Merkle tree (SMT) over a
256-bit keyspace requires traversing a path of depth 254: approximately
254 hash operations per proof path inside the STARK circuit.

An IMT at depth 32 instead supports non-membership proofs via finding
two adjacent values L and R where L.next = R and L < nullifier < R,
proving the nullifier is absent. The in-circuit path length is 32
hash operations: approximately 8x fewer per path than the SMT
approach.

This is not cosmetic. It directly reduces proving cost per fragment.

Non-membership proof: find the predecessor node L and successor node R
in the IMT's sorted structure where L.value < target < R.value and
L.next_index = R's index, via a depth-32 tree path. This proves target
is absent from the set.

The IMT root appears in every block header as nullifier_root_pre and
nullifier_root_post.

Hard consequence

The nullifier set is permanent consensus memory. It cannot be pruned
without changing the cryptographic assumptions. This is Specter's
dominant long-term state growth burden. At target throughput:

  Year 1:   ~8.5 GB of nullifier data
  Year 5:   ~42 GB
  Year 10:  ~84 GB

Plus ~30% IMT structural overhead. This is the storage cost Specter
pays for eliminating authorization artifacts. Nullifiers at 32 bytes
each are substantially cheaper than signatures at 3,309 bytes each.


/**** Block Structure ****/

A Specter block contains:

  Block:
    header:
      version               4 bytes
      height                8 bytes
      timestamp             8 bytes
      prev_hash            32 bytes
      utxo_root_pre        32 bytes   MMR root before this block
      utxo_root_post       32 bytes   MMR root after this block
      nullifier_root_pre   32 bytes   IMT root before this block
      nullifier_root_post  32 bytes   IMT root after this block
      diff_root            32 bytes   commitment to state_diff
      proof_commitment     32 bytes   commitment to aggregated proof
      epoch_cert_hash      32 bytes   H(0x00..0) if no epoch boundary
      min_anchor_height     8 bytes   lowest anchor_height among all
                                      included fragments; used by
                                      aggregated proof public inputs
                                      to bound the anchor window check
      coinbase_address     32 bytes   miner's declared receive address
      producer_address     32 bytes   block producer's receive address
      nonce                 8 bytes
      difficulty_target     4 bytes
      ────────────────────────────────
      Header total:        360 bytes

    aggregated_block_proof   ~50-200 KB  (ephemeral, pruned after
                                          PROOF_RETENTION_WINDOW)

    state_diff:
      new_nullifiers[]     32 bytes each
      new_outputs[]       104 bytes each
      fee_kernels[]        32 bytes each

Header field notes:

  epoch_cert_hash:
    At non-epoch-boundary blocks: set to the zero hash (32 zero bytes).
    At epoch boundary blocks: set to the hash of the EpochCertificate
    that covers the closing epoch. See Epoch Proofs section.

  proof_commitment:
    Commitment to the aggregated proof blob, allowing proof integrity
    verification before the proof itself is downloaded.

  diff_root:
    Commitment to the state_diff payload. Enables diff validation
    without downloading the full proof.

  min_anchor_height:
    The minimum anchor_height across all fragments included in this
    block. The aggregated proof's public inputs bind to this value,
    allowing verifiers to confirm that all included fragments were
    within their valid anchor window at the time of block assembly
    (i.e., min_anchor_height >= height - ANCHOR_WINDOW). An empty
    block sets min_anchor_height = height.

  coinbase_address:
    The miner's receive address for the block reward and miner's
    fee share. This is a standard 32-byte address derived from the
    miner's wallet using the address model (see Authorization Without
    Public-Key Signatures).

  producer_address:
    The block producer's receive address for the producer's fee
    share. Nodes use coinbase_address and producer_address together
    to reconstruct and verify both coinbase output commitments in
    the state_diff.

The pre/post root pairs make state-transform semantics auditable:
any node can independently verify declared post-roots follow from
declared pre-roots by applying the included state_diff.


/**** Consensus Validation ****/

To validate a block, a node performs in order:

  1. Header sanity
     Check version, height continuity, timestamp validity,
     prev_hash linkage.

  2. Proof-of-work
     pow_hash = H(H(header || nonce))     (double-BLAKE3)
     Valid iff pow_hash < difficulty_target.

  3. Difficulty adjustment
     Verify ASERT difficulty against the canonical anchor parameters.

  4. Proof commitment integrity
     Verify aggregated proof blob matches proof_commitment.

  5. Aggregated proof verification
     Verify aggregated proof against:
       utxo_root_pre, utxo_root_post,
       nullifier_root_pre, nullifier_root_post,
       diff_root, anchor_height.

  6. State-diff application
     Apply new_nullifiers to the IMT.
     Apply new_outputs to the MMR.
     Apply fee accounting to coinbase.

  7. Root consistency
     Check locally recomputed post-roots match header commitments.

  8. Monetary correctness
     total_fragment_fees = sum(fee_amounts in state_diff)
     reward = scheduled_reward(height)
     producer_fee     = (total_fragment_fees * BLOCK_PRODUCER_FEE_SHARE) / 100
     miner_fee        = total_fragment_fees - producer_fee
     producer_subsidy = (reward * BLOCK_PRODUCER_COINBASE_SHARE) / 100
     miner_reward     = reward - producer_subsidy
     miner_coinbase_value    = miner_reward + miner_fee
     producer_coinbase_value = producer_subsidy + producer_fee
     Verify both coinbase outputs match expected values.
     Verify ProducerCoinbase is present when reward > 0.

  9. Same-block spend rule
     No output created in this block is also consumed in this block.

  10. Epoch certificate check
      If height mod EPOCH_LENGTH != 0: verify epoch_cert_hash is zero.
      If height mod EPOCH_LENGTH == 0:
        - epoch_cert_hash must be either a valid cert_id or the
          canonical failure marker H("specter.epoch-fail" || epoch_index).
        - If epoch_cert_hash is a cert_id: verify the referenced
          EpochCertificate is locally available and valid.
        - If epoch_cert_hash is a failure marker: the epoch is
          uncertified; EPOCH_DEBT_PENALTY applies to all coinbase
          outputs in that epoch (enforced at spend time via check 8a).
        - epoch_cert_hash being any other value is invalid.

All ten checks must pass. Proof verification (step 5) is one STARK
verification call: approximately 5-20 ms, independent of fragment
count. This is the key verification asymmetry.


/**** State Diff Format ****/

The canonical state_diff is a flat list of state effects:

  state_diff:
    new_nullifiers[]
    new_outputs[]
    fee_kernels[]

Where:

  fee_kernel = H(
    fragment_id ||
    LE64(fee)   ||
    "specter.fee"
  )

The canonical block record does not preserve the explicit mapping
between consumed nullifiers and created outputs, nor which outputs
belonged to the same user action. That mapping exists only inside
the proof during the proof window. After proof expiry, the canonical
chain preserves state effects, not causal traces.

This is intentional. It is the source of spending unlinkability at
the canonical chain level.

Causal hint objects (optional, non-consensus):

Nodes MAY retain non-consensus CausalHint records for indexing:

  CausalHint:
    fragment_id           32 bytes
    consumed_output_ids[] 32 bytes each
    created_output_ids[]  32 bytes each
    fee                    8 bytes
    memo_commitment       32 bytes  (optional, opaque)

CausalHints do not affect state validity. They may be pruned by any
node without affecting consensus correctness. They help wallets,
explorers, and accounting tools reconstruct spend linkage without
requiring indefinite proof retention.


/**** Coinbase Outputs ****/

Every block produces two coinbase outputs that distribute the
scheduled block reward and fragment fees between the miner and the
block producer according to the consensus fee and coinbase split rule.

Split computation:

  total_fragment_fees = sum(fee_amount for all fragments in block)
  reward              = scheduled_reward(height)

  producer_fee     = (total_fragment_fees * BLOCK_PRODUCER_FEE_SHARE) / 100
  miner_fee        = total_fragment_fees - producer_fee
  producer_subsidy = (reward * BLOCK_PRODUCER_COINBASE_SHARE) / 100
  miner_reward     = reward - producer_subsidy

Miner coinbase output:

  MinerCoinbase:
    address     32 bytes   coinbase_address from block header
    value        8 bytes   miner_reward + miner_fee
    rho         32 bytes   H("specter.coinbase.miner" || height || prev_hash)

Producer coinbase output (always present when reward > 0):

  ProducerCoinbase:
    address     32 bytes   producer_address from block header
    value        8 bytes   producer_subsidy + producer_fee
    rho         32 bytes   H("specter.coinbase.producer" || height || prev_hash)

Both coinbase rho values are deterministically derived from consensus-
visible data. This prevents rho grinding attacks and makes both
coinbase output commitments fully deterministic given the block header.

Coinbase output commitments:

  miner_coinbase_cm = H(
    coinbase_address  ||
    LE64(miner_value) ||
    miner_rho         ||
    "specter.output"
  )

  producer_coinbase_cm = H(
    producer_address    ||
    LE64(producer_fee)  ||
    producer_rho        ||
    "specter.output"
  )

Both coinbase outputs use exactly the same output model as all other
outputs. They enter the UTXO MMR as standard output records at 104
bytes each. Canonical ordering in new_outputs[]: miner coinbase first,
producer coinbase second (if present), then all other new_outputs.

Coinbase spend constraints:

  COINBASE_MATURITY = 100 blocks

A coinbase output (miner or producer) cannot be consumed until block
height (creation_height + COINBASE_MATURITY). This is a hard consensus
rule. The block producer MUST reject any fragment that references a
coinbase output whose creation height is within the maturity window.

Consensus check for coinbase validity (additional check added to
block validation step 8):

  8a. Coinbase maturity
      For each consumed output in new_nullifiers[]: if the output is
      a coinbase output (miner or producer) created at block height
      h_c in epoch e:
        - If epoch e is certified (epoch_cert_hash != failure marker):
            maturity height = h_c + COINBASE_MATURITY
        - If epoch e is uncertified (epoch_cert_hash == failure marker):
            maturity height = h_c + COINBASE_MATURITY + EPOCH_DEBT_PENALTY
      Reject if the current block height < maturity height.

Identifying coinbase outputs: miner coinbase has rho derived as
H("specter.coinbase.miner" || height || prev_hash); producer coinbase
has rho derived as H("specter.coinbase.producer" || height || prev_hash).
Nodes index these derivations at creation time.

The miner and block producer declare their addresses in the block
header:

  coinbase_address   32 bytes
  producer_address   32 bytes

Consensus nodes verify both coinbase output commitments against the
header-declared addresses, computed rho values, and the fee split
computation.


/**** Epoch Proofs ****/

This is the architectural component that separates Specter from chains
that permanently discard authorization history without a replacement
mechanism.

The problem with simple proof pruning

If per-block proofs are discarded without replacement, a new node
syncing from genesis after the proof window cannot independently
verify that historical state transitions were correctly authorized.
It must inherit the canonical chain by social consensus.

This is the "weak subjectivity" tradeoff that Specter v1 explicitly
closes with epoch proofs.

Epoch structure

  EPOCH_LENGTH = 2,016 blocks   (~2.8 days at 120-second block time)

Every EPOCH_LENGTH blocks, the network requires an EpochCertificate
that covers the just-completed epoch.

EpochCertificate format:

  EpochCertificate:
    epoch_index              8 bytes
    epoch_start_height       8 bytes
    epoch_end_height         8 bytes
    pre_state_root          32 bytes   state root at epoch_start_height - 1
    post_state_root         32 bytes   state root at epoch_end_height
    header_chain_root       32 bytes   MMR root of all block headers in epoch
    block_proof_cm_root     32 bytes   Merkle root of all proof_commitments
                                       from block headers in the epoch
    accumulated_difficulty  32 bytes   LE256: sum of per-block difficulties
                                       across the epoch
    epoch_proof             variable   (~50-200 KB compressed)
    cert_id                 32 bytes   H(all above fields except epoch_proof)

EpochCertificate binding properties:

  An EpochCertificate is not an alternative chain history object.
  It is a succinct validity attestation over a PoW-anchored header
  sequence that is already canonically selected by the heaviest-chain
  rule. Every field in the EpochCertificate is bound to the specific
  block header sequence it covers:

  - epoch_start_height and epoch_end_height fix the exact block range.
  - header_chain_root commits to every block header in the range. An
    epoch certificate for a different header sequence would have a
    different header_chain_root.
  - block_proof_cm_root commits to every block's proof_commitment
    field. A forged epoch cert that accepts different state transitions
    than the ones in the canonical headers would need to forge the
    proof_commitment values in the headers — which are themselves
    committed to by the block's PoW hash.
  - accumulated_difficulty commits to the total PoW work in the epoch.
    An alternative epoch cert over a lighter chain would have a lower
    accumulated_difficulty, making substitution detectable by any node
    that has the canonical header chain.

What the epoch proof proves:

  "There exists a sequence of valid blocks with headers exactly
   matching header_chain_root, with block proofs exactly matching
   block_proof_cm_root, such that applying all their authorized state
   transitions transforms pre_state_root into post_state_root with
   total accumulated work equal to accumulated_difficulty."

The epoch proof is not a standalone historical claim. It is a proof
over a specific PoW header sequence that has already been selected
as canonical. A validator of the EpochCertificate verifies both the
epoch proof and that the header_chain_root matches the block headers
they have already accepted from the canonical chain.

This is a recursive compression of all per-block proofs in the epoch
into one compact epoch-level validity certificate that is inseparable
from the PoW-selected header sequence it covers.

Production mechanism

Epoch proofs use the same recursive STARK aggregation infrastructure
as block aggregation, applied at epoch scale. The epoch prover takes:

  - all retained per-block proofs for the epoch
  - all block headers for the epoch
  - all state diffs for the epoch

and produces one EpochCertificate via recursive compression.

This is not speculative. Mina Protocol has operated a constant-size
succinct blockchain proof from genesis since 2020 using exactly this
approach. SP1's recursive shard aggregation is documented as capable
of "proving a range of blocks."

Consensus requirement

An epoch boundary block (height mod EPOCH_LENGTH == 0) is not fully
finalized until its EpochCertificate is observed and validated.

The EpochCertificate must be produced within:

  EPOCH_FINALIZATION_WINDOW = 144 blocks   (~4.8 hours)

Epoch finalization modes

Specter v1 specifies two epoch finalization outcomes with distinct
economic consequences. There is no soft middle path.

Mode A — Certified epoch (normal operation):

  A valid EpochCertificate appears within EPOCH_FINALIZATION_WINDOW.
  The epoch boundary block's epoch_cert_hash is set to cert_id.
  All coinbase outputs from blocks in that epoch become eligible to
  mature after COINBASE_MATURITY blocks, counted from their block
  height.
  Historical bootstrap nodes can fully trustlessly verify that epoch.

Mode B — Uncertified epoch (fault condition):

  No valid EpochCertificate appears within EPOCH_FINALIZATION_WINDOW.
  The epoch boundary block's epoch_cert_hash records the failure
  marker: H("specter.epoch-fail" || epoch_index).

  Economic consequences (consensus-enforced, not social):

    1. Coinbase reward deferral.
       All coinbase outputs from blocks in that epoch have their
       COINBASE_MATURITY extended by EPOCH_DEBT_PENALTY blocks:

         effective_maturity = COINBASE_MATURITY + EPOCH_DEBT_PENALTY
         EPOCH_DEBT_PENALTY = EPOCH_LENGTH   (2,016 blocks, ~2.8 days)

       A miner who built a block in an uncertified epoch must wait
       an additional 2.8 days before spending their reward. This
       is enforced by consensus validation check 8a (coinbase
       maturity), which reads epoch_cert_hash to determine which
       maturity rule applies.

    2. Historical debt flag.
       The epoch boundary block is permanently flagged as uncertified
       in local node state. Historical bootstrap nodes that encounter
       a flagged epoch MUST obtain the block headers and state diffs
       for that epoch but cannot obtain an EpochCertificate for it.
       They fall back to verifying individual block state diffs against
       the pre/post state roots using the retained state commitments.
       This is computationally more expensive but still mechanically
       verifiable.

    3. No trust waiver.
       Specter does not define a "weakly trusted" trust level. An
       epoch is either certified (Mode A) or it carries a historical
       debt flag (Mode B). There is no third category.

Epoch prover bounty

EPOCH_DEBT_PENALTY provides a stick (delayed coinbase for uncertified
epochs). The epoch prover bounty provides a carrot: a direct economic
reward for producing the EpochCertificate.

  EPOCH_PROVER_BOUNTY_SHARE = 5   (percent of total epoch fees)

At each epoch boundary, the total fragment fees collected across all
blocks in the closing epoch are computed:

  epoch_total_fees = sum(total_fragment_fees for all blocks in epoch)
  epoch_prover_bounty = (epoch_total_fees * EPOCH_PROVER_BOUNTY_SHARE) / 100

The epoch prover bounty is funded by a small share of the fragment
fees that were already collected during the epoch. This share is
deducted from the miner's portion of the fee split (not from the
block producer's share), applied retroactively at epoch certification
time. The deduction is spread equally across all blocks in the epoch.

Bounty distribution:

The first valid EpochCertificate submitted within the
EPOCH_FINALIZATION_WINDOW earns the bounty. The epoch boundary block
includes a special bounty output:

  EpochProverBounty:
    address   = epoch_prover_address (declared in the EpochCertificate)
    value     = epoch_prover_bounty
    rho       = H("specter.epoch-bounty" || epoch_index || cert_id)

The epoch_prover_address is a new field in the EpochCertificate:

  EpochCertificate (extended):
    ...existing fields...
    epoch_prover_address  32 bytes   prover's receive address

The bounty output follows the standard output model and is subject to
COINBASE_MATURITY. It is appended to the epoch boundary block's
new_outputs[] after the coinbase outputs.

Economic rationale:

At target throughput (500 fragments/block, ~0.001 SPEC average fee),
a 2,016-block epoch generates approximately:

  epoch_total_fees ~ 2,016 x 500 x 0.001 = ~1,008 SPEC
  epoch_prover_bounty ~ 1,008 x 0.05 = ~50.4 SPEC

This is sufficient to cover the GPU cost of recursive epoch proof
generation (estimated at $5-50 per epoch at current hardware prices)
while representing only 5% of total epoch fee revenue.

If the epoch is uncertified (no valid EpochCertificate produced), the
bounty is not paid and the fees remain with the original recipients.
The combination of bounty (positive incentive) and EPOCH_DEBT_PENALTY
(negative incentive) creates strong economic pressure for timely
certificate production from both directions.

Rationale for economic enforcement over social consensus:

Epoch certification is not ancillary infrastructure. It is the
mechanism that prevents trustlessness from degrading over time. A
protocol that says "please produce epoch proofs; if you don't, we'll
note it" has no enforcement lever. A protocol that says "your block
rewards are locked longer if your epoch is uncertified AND someone
else gets paid for producing the certificate" creates dual economic
incentives: miners are motivated to facilitate certification (to
avoid the penalty), and independent provers are motivated to produce
certificates (to earn the bounty).

The EPOCH_DEBT_PENALTY targets the operators who received block
rewards during the uncertified epoch, not subsequent operators.
The epoch prover bounty targets any capable participant, creating
an open market for certificate production independent of block
production. This is a targeted, proportionate economic signal, not
a global chain stall.

Storage consequence

After EPOCH_FINALIZATION_WINDOW blocks:

  - per-block proof blobs for the epoch: DISCARDED
  - EpochCertificate: RETAINED PERMANENTLY (~50-200 KB per epoch)

For EPOCH_LENGTH = 2,016 blocks: approximately one EpochCertificate
every 2.8 days = ~130 epoch certs per year = ~13-26 MB of permanent
epoch certificate storage per year.

This is a dramatically smaller permanent storage burden than
retaining all per-block proofs, while preserving full trustless
historical verifiability.

Historical bootstrapping with epoch proofs

A new node syncing from genesis:

  1. Downloads all block headers (360 bytes each, ~25 MB/year)
  2. Downloads all EpochCertificates (~13-26 MB/year)
  3. Verifies each EpochCertificate covers its epoch's headers
  4. Trusts only: PoW chain, epoch proof soundness, current state

This provides near-Bitcoin-level historical trustlessness at a
fraction of the storage cost.


/**** Proof Availability and Expiry ****/

Per-block proofs are ephemeral by design.

  PROOF_RETENTION_WINDOW = 1,440 blocks   (~48 hours at 120-second
                                            block times)

During the proof retention window:

  - nodes can directly verify any recent block's proof
  - nodes can verify per-fragment authorization validity
  - light clients can verify recent state evolution

Before a proof expires, the epoch proof for its epoch must be
produced (see Epoch Proofs). Once the EpochCertificate is finalized
and stored, the per-block proof is redundant for historical purposes.

After the proof retention window:

  - the per-block aggregated proof is discarded
  - the block retains: header, state_diff, state commitments
  - historical authorization verifiability is provided by the
    EpochCertificate covering that block's epoch

Storage accounting:

  Permanent per-block storage:
    Header:       360 bytes
    State diff:   ~156 KB per block at target throughput
    Epoch cert:   ~100 KB amortized per block (50-200 KB / 2016 blocks)
    Total:        ~156 KB per block permanent

  Temporary (pruned after 48h):
    Block proofs: ~50-200 KB per block


/**** Consensus: Proof-of-Work ****/

Block production is permissionless and Sybil-resistant via PoW.

PoW function:

  pow_hash = H(H(header || nonce))   (double-BLAKE3)

A block is valid if pow_hash < difficulty_target.

BLAKE3 is used for PoW because it is fast to evaluate (3-8 GB/s
single-threaded), simple to implement correctly, requires no
special circuit design, and produces 256-bit outputs.

Why proof-of-work

Specter's authorization layer already carries bounded-historical-
trust semantics from proof expiry. Adding PoS on top would introduce
a second weak-subjectivity layer (stake-history checkpointing) on top
of the authorization-history epoch assumption. This would compound
trust assumptions rather than reduce them.

PoW avoids this. Historical chain selection is determined by the most
accumulated work, independent of stake history. This aligns with
Specter's goal of minimizing trust assumptions after the epoch window.


/**** Difficulty Adjustment ****/

Specter uses ASERT (aserti3-2d), deployed on Bitcoin Cash since the
Axion hard fork on November 15, 2020.

ASERT is smooth and responsive, without the oscillation pathologies
of older retarget designs under variable hashrate. Kawaguchi, Komiyama,
and Noda (International Economic Review, 2025, DOI: 10.1111/iere.70060)
prove that ASERT is stable under any positive miner elasticity and
characterize it as stochastic gradient descent on the difficulty
surface.

Consensus parameters:

  BLOCK_TIME_TARGET  =  120 seconds
  ASERT_HALFLIFE     =  172,800 seconds   (48 hours)

Canonical update form:

  next_target = anchor_target * 2^(
    (elapsed_time - BLOCK_TIME_TARGET * (height_delta + 1))
    / ASERT_HALFLIFE
  )

All consensus arithmetic is integer-only. Floating-point is forbidden
in consensus code.


/**** Emission ****/

Specter uses smooth exponential decay with a perpetual tail emission.

All reward values are denominated in base units. Define:

  SPEC_BASE_UNITS   = 100,000,000   (1 SPEC = 10^8 base units)
  INITIAL_REWARD    = 50 * SPEC_BASE_UNITS  = 5,000,000,000 base units
  DECAY_NUMERATOR   = 999999
  DECAY_DENOMINATOR = 1000000
  TAIL_REWARD       = 60,000,000 base units   (0.6 SPEC)

Integer-safe reward computation:

The floating-point formula reward(h) = max(floor(50 * 0.999999^h), 0.6)
is not safe for consensus code. The canonical integer implementation
uses a geometric decay computed via repeated multiplication, cached
in a lookup table precomputed at node startup:

  reward_table[0] = INITIAL_REWARD
  reward_table[h] = (reward_table[h-1] * DECAY_NUMERATOR) /
                     DECAY_DENOMINATOR       (integer division)
  if reward_table[h] < TAIL_REWARD:
      reward_table[h] = TAIL_REWARD

All arithmetic is 64-bit unsigned integer. The division is integer
(floor) division. This recurrence is equivalent to the floating-point
formula but avoids floating-point entirely. Implementations MUST use
this recurrence, not the floating-point form.

TAIL_HEIGHT is the smallest h where reward_table[h] == TAIL_REWARD.
This is approximately h = 3,912,023 under the recurrence above.

Intended properties:

  - smooth decline with no halving cliffs
  - perpetual block reward floor for long-term security budget
  - reduced fee-only dependence over time

Schedule:

  Tail floor reached:  ~height 3,912,023 (~14.9 years at 720 blk/day)
  Pre-tail supply:     ~49.0 million SPEC
  Tail emission:       0.6 SPEC/block (~157,680 SPEC/year)
  Year-10 inflation:   ~2.0%
  Year-20 inflation:   ~0.3%

Launch policy: no premine, no founder allocation, no ICO, no
treasury. If the protocol cannot stand on its own economics, it
should not exist.


/**** Proof-Aware Fee Market ****/

Not all fragments impose equal cost on block producers. A fragment
with 4 consumed outputs and 4 created outputs costs more to prove and
aggregate than a standard 1-in/2-out spend. A fee model that ignores
this creates systematic under-pricing of heavy fragments and
over-admission of resource-intensive work.

Weight classes establish the cost basis:

  W1:  1 consumed output, 1-2 created outputs
       weight_cost = 1.0 unit

  W2:  2 consumed outputs, 1-3 created outputs
       weight_cost = 1.8 units

  W3:  3-4 consumed outputs, 1-4 created outputs
       weight_cost = 3.0 units

Effective fee score (used for block producer ordering):

  fee_score = fee / weight_cost

Block producers select fragments in descending fee_score order,
tie-broken by ascending fragment_id.

Minimum fee per weight class is a local policy parameter, not a
consensus-enforced constant. Nodes SHOULD enforce a positive minimum
fee for each class to prevent spam.

Fee kernels in the state_diff:

  fee_kernel = H(fragment_id || "specter.fee")

The fee amount is not separately committed inside fee_kernel because
fragment_id already commits to the fee (fragment_id's preimage includes
LE64(fee)). Including LE64(fee) again in fee_kernel would double-commit
the fee with no additional security property. The fee_kernel serves
as a compact per-fragment handle for coinbase accounting without
introducing redundant data.

The raw fee amount MUST still be included in the state_diff as an
explicit field alongside each fee_kernel for supply auditing and
coinbase verification:

  state_diff fee entry:
    fee_kernel   32 bytes   H(fragment_id || "specter.fee")
    fee_amount    8 bytes   LE64(fee)

This separation keeps fee_kernel compact while making fee_amount
directly readable without deriving it from the fragment_id preimage.


/**** Cryptographic Primitive Split ****/

Specter uses two hash families with strictly separated roles.

This separation is required, not aesthetic.

General-purpose hashes such as BLAKE3 are efficient on CPUs but
require thousands of algebraic constraints per invocation inside a
STARK circuit. Using BLAKE3 for every in-circuit hash relation would
make per-fragment proving costs prohibitive.

Poseidon2 was designed for in-circuit use. It achieves low constraint
counts while maintaining adequate security margins for proof soundness.

The two primitives serve different security budgets (WSC-128 and
PSC-64 respectively), as defined in the Security Classes section.


/**** External Hashing: BLAKE3 ****/

BLAKE3 is used for all external protocol operations:

  - wallet root derivation
  - per-output secret derivation
  - address derivation
  - output commitment computation
  - nullifier derivation
  - transition binding and auth tags
  - fragment IDs, intent IDs, witness bundle IDs
  - block header hashing and PoW
  - epoch certificate hashing
  - all out-of-circuit MMR and IMT node hashing

BLAKE3 produces 256-bit outputs. It runs at approximately 3-8 GB/s
single-threaded on modern x86 CPUs with SIMD (hardware-dependent;
~3 GB/s without wide SIMD, ~8 GB/s with AVX2/AVX-512 on AMD Zen 4
or Intel Cascade Lake).

BLAKE3 supports built-in domain separation via:

  - standard hashing mode
  - keyed hashing (`keyed_hash`, 32-byte key)
  - key derivation (`derive_key`, application context string)

Specter uses the `derive_key` mode with string tags for all
domain-separated hash invocations.

Post-quantum security:

  Preimage:   ~128-bit PQ resistance (Grover: 2^256 → ~2^128)
  Collision:  ~85-bit PQ resistance  (BHT:    ~2^(256/3))

The 128-bit preimage figure governs wallet-secret security (WSC-128).
The 85-bit collision figure governs output commitment binding and
should not be confused with preimage hardness.


/**** In-Circuit Hashing: Poseidon2 ****/

Poseidon2 is used only inside the STARK proving system.

Canonical parameters (as deployed in Plonky3/SP1):

  Field:             BabyBear  (p = 2^31 - 2^27 + 1 = 2,013,265,921)
  State width:       16
  Rate:              8 field elements
  Capacity:          8 field elements (248 bits effective)
  S-box:             x^7
  External rounds:   8
  Internal rounds:   13
  Classical security target: 128 bits

In-circuit role:

  - witness-column commitments in the proving relation
  - IMT and MMR path verification inside the circuit
  - internal compression and consistency constraints

Constraint cost:

The exact per-invocation AIR constraint count for Poseidon2 over
BabyBear in Plonky3 depends on implementation choices. Documented
estimates place Poseidon2 at approximately 150-200 algebraic
constraints per invocation for a 2-field-element hash (Plonky3/SP1
production parameters). A precise figure requires reference to the
Plonky3 AIR source.

Security:

  Classical security:      128 bits (target)
  PQ preimage security:    ~64 bits (Grover halving on 248-bit capacity)
  PQ collision security:   ~42 bits (BHT on 248-bit capacity)

The ~64-bit PQ preimage security governs proof-system soundness
(PSC-64). It is sufficient for the soundness requirement but does not
achieve wallet-level secret hardness. The distinction is explicit and
intentional in Specter's design (see Security Classes).

Cryptanalysis status:

ePrint 2025/937 (Sanso and Vitto) and ePrint 2025/950 (Zhao and Ding)
introduce Graeffe-transform-based attacks on Poseidon2. Both papers
target reduced-round instances from the Ethereum Foundation's
cryptanalytic bounty program. Full-round Poseidon2 over BabyBear as
deployed in Plonky3/SP1 is not broken by these papers. The papers
confirm the security margin by successfully attacking only reduced-
round variants, which is the intended behavior of the bounty program.


/**** STARK Proof System ****/

Specter targets the Plonky3 FRI-based STARK proving system over the
BabyBear field.

Why STARKs

  - no trusted setup ceremony
  - recursive composability
  - soundness based on hash/proximity assumptions rather than
    public-key hardness
  - no known Shor-style quantum attack

A chain whose authorization layer is entirely ZK-mediated cannot
depend on a one-time trusted ceremony: a compromised ceremony allows
forging valid proofs for false statements. Specter uses a transparent
proof system with no ceremony.

BabyBear field

  p = 2^31 - 2^27 + 1 = 2,013,265,921
  Two-adicity: 2^27, supporting NTT domains up to ~134 million rows
  Field multiplication: ~0.4 cycles/operation with AVX-512 SIMD

Plonky3 is production-deployed in SP1 (Succinct Labs), which is
integrated into Optimism's OP Stack as OP Succinct. Recursive proof
composition is live in production: SP1's verifier circuit is itself
run inside SP1, allowing constant-size aggregated proofs.

GPU acceleration is available via the ICICLE library (CUDA, Metal,
CPU backends), providing approximately 3-7x speedup over CPU for
NTT-dominated operations.

Performance targets (engineering estimates, not consensus assumptions):

  Per-fragment proof generation:
    CPU:   ~1-5 seconds
    GPU:   ~0.5-2 seconds

  Aggregated block proof:
    Size:          ~50-200 KB
    Verification:  ~5-20 ms (independent of fragment count)

  Block aggregation (GPU-assisted):
    Target under 120 seconds for target block fullness (~500 fragments)

Zero-knowledge for witness values is achieved via witness polynomial
masking (Haböck and Al Kindi, ePrint 2024/1037): random low-degree
polynomials are added to witness trace columns, vanishing on the trace
domain. This prevents witness value recovery from the proof transcript.


/**** Post-Quantum Fiat-Shamir: Formal Status ****/

FRI-based STARKs and their Fiat-Shamir transformations have formal
post-quantum security in the QROM.

Block, Garreta, Liu, Losher, Magri, Riembau, and Vidick established
round-by-round soundness for FRI and several related protocols
(including ethSTARK and Batched FRI) in "Fiat-Shamir Security of FRI
and Related SNARKs" (ASIACRYPT 2023, ePrint 2023/1071). Via the BCS
transformation, any round-by-round sound IOP yields a SNARG that is
secure against quantum adversaries in the QROM.

This resolves the concern present in earlier drafts of this document.
Specter's Fiat-Shamir security claim for FRI-based STARKs has a
formal proof. Specter therefore states:

  Specter's FRI-based STARK authorization is post-quantum secure
  in the quantum random oracle model, per Block et al. 2023.

The remaining honest caveat is implementation-level:

  - formal proofs cover the protocol class; actual security depends
    on the specific AIR being correctly specified
  - the Plonky3 implementation has not received a formal AIR
    machine-checked security proof
  - a full formal audit of the spend circuit AIR is required before
    any production mainnet deployment claim

This is a practical implementation gap, not a theoretical gap.


/**** Security Claims ****/

Claim 1: Spend Authorization Soundness

No PPT or QPT adversary can produce a valid spend proof for an output
they do not own without either:

  - knowing a valid witness consistent with the output and transition,
  - breaking the proving relation soundness (STARK soundness), or
  - breaking BLAKE3 preimage resistance at WSC-128 level.

Claim 2: Double-Spend Resistance

No valid chain history can consume the same spendable output twice.
This reduces to: nullifier uniqueness, IMT non-membership enforcement
per fragment, and correct recursive state application in the epoch
proof stack.

Claim 3: Replay Resistance

A valid fragment authorization cannot be replayed outside its anchor
window or against a different transition without reproving. This
reduces to: transition_hash binding, auth_tag derivation, anchor-
window semantics, and block-assembly enforcement.

Claim 4: State Evolution Soundness

A valid block proof certifies that: every included fragment is valid,
the state_diff exactly matches the accepted effects, and the declared
post-roots are correctly computed from the accepted fragment set.

Claim 5: Historical Verifiability

A new node can trustlessly verify the entire chain history using:
PoW header chain, EpochCertificates, and current accumulated state.
Trust assumption: STARK soundness of the epoch proof stack plus
correct AIR binding of epoch proofs to header_chain_root and
block_proof_cm_root (see Open Problem 1).

Claim 5a: EpochCertificate Non-Substitutability

An EpochCertificate cannot be substituted for a different one covering
the same epoch without either: (a) producing a valid STARK proof over
a different header_chain_root, which would immediately fail against
the retained canonical block headers; or (b) rewriting the canonical
block headers for that epoch, which requires a PoW chain rewrite of
equal or greater total difficulty than the canonical chain. This
reduces EpochCertificate substitution to a standard 51% attack, not
a distinct threat class created by epoch proof compression.

Claim 6: Quantum-Surface Reduction

Specter's permanent canonical chain does not expose public-key
authorization artifacts vulnerable to Shor's algorithm. The
authorization surface is hash-preimage-based (WSC-128 via BLAKE3)
and STARK-soundness-based (PSC-64 via Poseidon2/FRI).


/**** Security Dependencies ****/

Specter's security depends on all of the following. Any serious
description of the system must name them all.

  1. BLAKE3 preimage resistance at 256-bit output (WSC-128)
  2. Poseidon2 binding security at deployed BabyBear parameters (PSC-64)
  3. Correct AIR specification of the spend circuit
  4. STARK soundness of Plonky3 (FRI-based, BabyBear field)
  5. QROM security of FRI's Fiat-Shamir transformation (Block et al.
     2023, ePrint 2023/1071)
  6. Recursive proof composition correctness
  7. Epoch proof correctness, including correct AIR binding of
     epoch proofs to header_chain_root and block_proof_cm_root
  8. Correct implementation of all consensus rules, including
     epoch finalization modes, EPOCH_DEBT_PENALTY, epoch prover
     bounty, and fee distribution
  9. Correct implementation of the wallet derivation scheme


/**** Adversarial Model ****/

Specter assumes adversaries may attempt:

  - witness forgery (forge a spend proof without the owner secret)
  - proof forgery (produce a valid proof for a false state transition)
  - nullifier replay (submit a spend for an already-spent output)
  - stale-fragment griefing (force excessive reproving via mempool attacks)
  - intent flooding (broadcast SpendIntents without following proofs)
  - fragment withholding (block producer suppresses valid fragments)
  - proof-availability eclipse (isolate a node during proof window)
  - delegated-proving witness theft (steal witness material from prover)
  - hardware centralization capture at the block producer layer
  - epoch proof withholding (delay or prevent EpochCertificate creation)
  - long-range epoch certificate substitution (requires simultaneous
    PoW chain rewrite for the target epoch; reduces to 51% attack)

The protocol addresses some of these directly and names others as
residual operational risks.


/**** Adversarial Failure Modes ****/

1. Stale-fragment griefing

Mitigations: anchor-window binding (ANCHOR_WINDOW = 8 blocks), anchor
rebasing (reattach attachment proof without reproving core ownership),
rolling witness cache (pre-fetched witnesses avoid latency at send
time), two-phase intent protocol with short INTENT_EXPIRY_WINDOW
(4 blocks) to limit spam-slot occupation. Not fully eliminated for
extreme contention.

2. Proof-availability eclipse

A node isolated during the proof retention window loses access to
recent validity evidence and cannot verify recent blocks directly.
A wallet isolated from witness bundle providers cannot generate
spend proofs regardless of cryptographic correctness.

Mitigations: block producer witness serving obligation (primary source
for newly created outputs), full node p2p witness serving obligation,
peer diversity, multi-source witness verification (minimum 3 sources
attempted), optional archival-proof retention by specialized nodes.
Witness serving is attached to already-incentivized roles (block
producers via fee split, full nodes via standard participation).
Residual risk: eclipse attacks on wallet peers remain a real
operational risk comparable to Bitcoin's block relay eclipse.

3. Epoch proof withholding

An attacker with sufficient block production influence attempts to
prevent EpochCertificates from being produced or gossiped.

Mitigations:
  - Any capable node can produce the EpochCertificate independently
    (not only the block producer); the epoch prover role is open to
    any participant with sufficient compute.
  - Epoch prover bounty (EPOCH_PROVER_BOUNTY_SHARE = 5% of epoch
    fees) provides direct positive incentive for independent provers
    to produce the certificate regardless of block producer behavior.
  - Epoch certificate gossip on a separate relay layer, independent
    of block gossip.
  - EPOCH_FINALIZATION_WINDOW of 144 blocks (~4.8 hours) provides
    substantial time for certificate production.
  - If withholding succeeds despite the bounty, the economic penalty
    falls on the operators who collected block rewards in the
    uncertified epoch: their coinbase outputs are subject to
    EPOCH_DEBT_PENALTY (additional 2,016 blocks of maturity delay).
  - The dual incentive (bounty + penalty) is consensus-enforced:
    no social agreement is required.

4. Block production centralization

Only GPU-equipped operators can aggregate proofs at scale within block
time.

Mitigations: block producer/miner role separation (keeps PoW
permissionless and decoupled from proving hardware), consensus-enforced
fee split and coinbase subsidy (block producers are explicitly
compensated from both fees and block reward), open proving
stack (Plonky3 is open source), deterministic fragment ordering (limits
MEV extraction), multiple block producers can compete per block height,
verification is O(1) and cheap for all nodes regardless. Concentration
at the block producer layer is accepted as a hardware-reality tradeoff;
it is non-critical because miners retain permissionless selection power.

5. Intent flooding

An attacker broadcasts SpendIntents without following through with
SpendFragments, occupying output locks in the mempool for up to
INTENT_EXPIRY_WINDOW blocks per intent.

Mitigations: short INTENT_EXPIRY_WINDOW (4 blocks vs ANCHOR_WINDOW 8),
one-active-intent-per-output-ID rule, per-peer rate limit
(MAX_INTENTS_PER_PEER_PER_BLOCK = 10), fee plausibility check on
intent admission. Residual risk: a well-connected attacker with many
peer connections could sustain a moderate intent flood; the per-peer
rate limit reduces effectiveness but does not fully eliminate it.

6. Delegated-proving witness theft

A wallet that delegates proving reveals witness material sufficient
to spend the selected outputs.

Mitigations: wallets MUST default to local proving, MUST require
explicit user consent for delegation, MUST warn that delegation is
equivalent to temporary spend authority transfer. Future versions may
support threshold proving and hardware enclave proving.

7. Nullifier-state growth

Long-term state burden pressures full-node decentralization.

Mitigations: efficient IMT design with batch insertion, nullifiers
are 32 bytes vs 3,309 for ML-DSA (100x smaller), intended nullifier
state tiering (hot IMT for recent epochs + cold epoch spentness
commitments for history), cut-through compression in archival nodes
(optional, non-consensus). Not fully solved at v1; tiering is the
intended v2 direction. The permanent cost is paid in spentness
memory, not signature artifacts.


/**** Node Architecture ****/

Three node classes:

1. Consensus full node

Maintains: full header chain, UTXO MMR, nullifier IMT, mempool,
recent block proofs (within PROOF_RETENTION_WINDOW), all
EpochCertificates, pruned historical proof blobs, CausalHints
(optional).

Can: fully validate current consensus, serve witness bundles to
wallets, serve accumulator inclusion proofs.

2. Block-producing node

A full node plus: recursive aggregation engine, GPU-class proving
hardware, block assembly and aggregation pipeline.

Can: produce valid BlockCandidates within the block time window,
serve witness bundles for included outputs.

3. Light client

Maintains: header chain, recent proof commitments, EpochCertificates,
wallet-local address scan state.

Can: verify recent block validity (~5-20 ms per block), verify entire
historical chain via EpochCertificates, receive funds, request witness
bundles from full nodes.

Cannot: prove spends without external witness bundles.


/**** Wallet Operation ****/

Rolling witness cache

The wallet maintains a spending-ready state for all owned outputs:

  - on output receipt: immediately request WitnessBundle from a full
    node or witness relay
  - periodically: refresh WitnessBundle as the accumulator advances
  - background: verify freshness, rebase anchor if needed

To receive

Wallet derives fresh address(n), publishes it, scans new state_diffs
for outputs matching its address set. No recipient interaction beyond
address publication.

To send (two-phase)

  Phase 1 (immediate):
    Select spendable output(s) from local UTXO state.
    Retrieve cached WitnessBundle(s).
    Compute transition_hash.
    Broadcast SpendIntent.
    Begin proof generation in background.

  Phase 2 (1-5 seconds later):
    Proof generation completes.
    Broadcast SpendFragment linking to intent_id.
    Monitor for confirmation.

If the anchor expires during Phase 2: rebase the attachment header
using a fresh WitnessBundle. The core ownership proof does not need
to be regenerated.

Wallet policies (MUST):

  - prove locally by default
  - require explicit user confirmation before delegated proving
  - maintain rolling witness cache for all owned outputs
  - support anchor rebasing without full reproving
  - track intent expiry and notify user of timeout

Wallet policies (SHOULD):

  - expose witness cache freshness status to user
  - warn on address reuse
  - cache fetched witnesses aggressively
  - support configurable scan horizon


/**** Throughput Target and Storage Model ****/

Throughput target

Specter v1 targets approximately 500 fragments per 120-second block
(~4.17 fragments/second). At ~2 outputs per fragment: ~1,000 outputs
per block. This is constrained by proof aggregation, not verification.

Verification is O(1) in fragment count regardless of block fullness.

Permanent canonical storage

  Per block:
    Header:             360 bytes   (296 + 32 coinbase + 32 producer)
    New nullifiers:     1,000 x 32  = 32,000 bytes
    New outputs:        1,002 x 104 = 104,208 bytes  (1,000 + 2 coinbases)
    Fee entries:          500 x 40  = 20,000 bytes   (32 kernel + 8 amount)
    Total per block:    ~156 KB

  Per day (720 blocks):
    ~112 MB/day of permanent canonical state growth

  Annual storage (canonical state only):
    ~41 GB/year

  EpochCertificates (permanent):
    ~130 certs/year x ~125 KB avg = ~16 MB/year

  Total permanent:     ~41 GB/year  (~112 MB/day + epoch certs)

Ephemeral storage (pruned per schedule):

  Block proofs:        ~50-200 KB x 1,440 blocks retained
                       ~72-288 MB at any time

  Intent relay:        small, short-lived

  Witness bundles:     small, refreshed frequently

Long-term nullifier set growth:

  Year 1:   ~8.5 GB
  Year 5:   ~42 GB
  Year 10:  ~84 GB
  Plus ~30% IMT structural overhead

Full-node storage estimate at year 5: ~200-300 GB total including
UTXO MMR, nullifier IMT, headers, output records, epoch certificates.

Specter does not claim zero authorization storage cost. It claims:
zero permanent canonical public-key authorization retention. The
storage cost is paid in nullifiers, not signatures.


/**** What Specter Does Not Provide ****/

  - amount privacy
  - address privacy
  - network-layer anonymity
  - indefinitely trustless history without epoch proofs
  - trustless history for uncertified epochs (Mode B: historical debt
    flag applies; state diffs are verifiable but no compact proof exists)
  - free or instant proving
  - script-based programmability
  - formally machine-checked AIR correctness (required before mainnet)
  - decentralization guarantees stronger than its hardware asymmetries
  - consensus-enforced witness serving (block producer obligation and
    full node p2p serving are protocol expectations, not slashable rules)


/**** Open Problems ****/

These are unresolved protocol pressure points, not future roadmap
items. Calling them "roadmap" obscures the fact that they are
constraints.

1. Formal AIR specification

The spend circuit AIR must be formally specified and machine-checked
against the intended consensus semantics before any production claim
is valid. No protocol that uses ZK for consensus-critical authorization
can skip this step. This is the primary pre-mainnet requirement.

2. Nullifier set growth

The nullifier set is permanent and is Specter's dominant long-term
state growth cost. Accumulators with O(1) non-membership proofs
(RSA-based, class-group-based) exist but depend on hardness assumptions
that are not post-quantum secure. Lattice-based hash-only accumulators
remain research-only. This is an open research problem with no clean
solution known in the post-quantum setting.

No architectural change eliminates the permanence requirement. Any
system that prevents double-spending must remember which outputs have
been spent. The question is what form that memory takes at runtime.

Intended direction: nullifier state tiering

Specter v1 does not implement state tiering, but identifies it as the
intended direction for reducing the live runtime burden.

The key distinction is:

  Spentness is permanent and non-negotiable.
  The live runtime structure that serves spentness does not have to be.

State tiering separates the nullifier set into two tiers:

  Hot tier — Live IMT:
    Contains nullifiers from the most recent N_HOT epochs.
    This is the fully live, in-memory IMT required for fragment
    verification. Non-membership proofs for fragment validation are
    always generated against the hot-tier IMT.
    Proposed N_HOT = 4 epochs (~11 days).

  Cold tier — Epoch spentness commitments:
    Nullifiers older than N_HOT epochs are not maintained as a live
    IMT in RAM. Instead, each epoch's nullifier set is compressed
    into an epoch spentness commitment:

      epoch_spentness_cm = H(
        "specter.spent" ||
        epoch_index      ||
        sorted_nullifiers_root   (Merkle root of the epoch's nullifiers)
      )

    The cold tier stores: one 32-byte commitment per epoch.
    At 130 epochs/year: ~4 KB/year of cold-tier commitments.

    Historical spentness queries (e.g., for a bootstrapping node)
    can be answered by requesting the full sorted nullifier list for
    a specific epoch from archival nodes, hashing it, and verifying
    against the stored epoch_spentness_cm.

Tiering does not reduce the total data that must exist somewhere.
It reduces what a live consensus node must hold in fast-access state.
A full archival node retains the complete nullifier history. A
consensus full node retains only the hot-tier IMT plus cold-tier
commitment hashes.

Fragment verification against the cold tier is not required. A
fragment's IMT non-membership proof targets the hot-tier root. If
the fragment's nullifier is not in the hot tier, it might still be
in the cold tier (a double-spend from an old output). The block
producer must check cold-tier membership for any fragment that passes
the hot-tier non-membership check:

  Cold-tier membership check:
    Request the sorted nullifier list for epochs [0, N-N_HOT] from
    an archival node. Verify membership using a binary search on the
    sorted list against the stored epoch_spentness_cm. This is an
    O(log M) lookup on cold data, required only for fragments that
    consume very old outputs. In practice, most fragments spend
    recent outputs and never touch the cold tier.

This is a concrete architectural direction, not a complete
specification. Full formalization requires: defining the exact
hot/cold boundary protocol, specifying how cold-tier checks are
delegated during assembly, and proving that the tiering does not
create double-spend vectors at the hot/cold boundary. This is
earmarked for v2 scope.

3. Block producer centralization

Competitive proof aggregation requires GPU-class hardware. Block
production may concentrate among well-equipped operators. The
consensus-enforced fee split and coinbase subsidy ensure block
producers are compensated even during low-fee network phases,
and deterministic fragment ordering limits MEV extraction, but hardware
concentration must be measured against deployed economics, not assumed
away.

4. Witness infrastructure resilience

Witness serving is part of spend availability, not ancillary
infrastructure. v1.3 addresses the primary incentive gap from earlier
drafts by attaching witness serving to the block producer role (which
is already compensated via the fee split) and defining full node
witness serving as a p2p protocol obligation.

What v1.3 specifies: wire format (WitnessBundle), self-verification
semantics, block-producer serving obligation for included outputs,
full-node p2p serving obligation, multi-source verification policy.

Residual risk: the block producer obligation covers newly created
outputs but does not guarantee long-term witness freshness for outputs
that remain unspent across many blocks. Full node p2p serving covers
this gap in practice, but the p2p obligation is not consensus-enforced
and relies on peer scoring. A well-resourced attacker who eclipses a
wallet from all full node peers could still delay or prevent spending.

This residual risk is comparable to Bitcoin's block relay eclipse
attack: a real operational risk but not a cryptographic failure mode.
Defense relies on peer diversity, not protocol enforcement.

5. Epoch prover coordination

Epoch certificate production is open to any participant with sufficient
compute, and the epoch prover bounty (EPOCH_PROVER_BOUNTY_SHARE = 5%
of epoch fees) provides direct economic motivation. The
EPOCH_FINALIZATION_WINDOW (144 blocks, ~4.8 hours) provides buffer.
Residual risk: if multiple provers race to produce the certificate,
only the first submission earns the bounty, potentially creating
wasted computation. A future version may introduce a commitment
scheme where provers signal intent before beginning computation.

6. Long-range attack surface

The attack model: after per-block proofs for epoch E are discarded
(after PROOF_RETENTION_WINDOW), could an adversary substitute a
forged EpochCertificate that attests to different state transitions
for epoch E than actually occurred?

Why this is not a new vulnerability — PoW binding argument:

An EpochCertificate is bound to a specific PoW header sequence via
header_chain_root and block_proof_cm_root (see EpochCertificate
binding properties). A forged EpochCertificate covering the same
epoch E would require producing valid STARK proof for a different
header_chain_root than what appears in the canonical headers for
epoch E.

The canonical headers for epoch E are not discarded. Block headers
are permanent (360 bytes each, ~24 MB/year). Any node that has the
canonical header chain for epoch E can verify that an EpochCertificate
claims header_chain_root consistent with those headers. A forged cert
over different state transitions would need to claim a different
header_chain_root, which would immediately fail verification against
the retained canonical headers.

Therefore: substituting a forged EpochCertificate is equivalent to
presenting a different header chain for epoch E. But selecting which
header chain is canonical is determined by the heaviest PoW chain
rule — not by the EpochCertificate. An attacker who wants to install
a forged EpochCertificate for epoch E must also have the heavier chain
for epoch E, which is a standard 51% attack constraint — not a new
attack surface created by epoch proof compression.

Stated precisely: EpochCertificate substitution without PoW rewrite
is not possible given the cert's header_chain_root and
block_proof_cm_root bindings and the permanent retention of block
headers. EpochCertificate substitution with PoW rewrite is a 51%
attack on the PoW chain — an existing, bounded, measurable threat
identical to standard chain reorg attacks.

Epoch certification does not introduce a new trust assumption.
It provides a compact proof over a header sequence whose selection
is already determined by PoW. Removing epoch proofs would not make
the system more secure; it would make historical verification
impossible for new nodes.

Secondary layer — social checkpoints (defense-in-depth):

At each epoch boundary, any participating node MAY publish:

  checkpoint_hash = H("specter.checkpoint" || epoch_index ||
                       epoch_boundary_block_hash)

signed with any off-protocol signature scheme. These are not
consensus-verified. They provide human-auditable external anchoring
for historical dispute resolution and are not required for security.

Formal analysis status:

The PoW binding argument above provides strong practical confidence.
The formal gap is: proving that the STARK soundness error does not
create an exploitable surface for constructing a valid epoch proof
over a modified header sequence that is still consistent with the
canonical header_chain_root. This requires a formal proof that the
epoch proof circuit correctly binds to header_chain_root and
block_proof_cm_root in the AIR specification. This is part of the
broader formal AIR correctness requirement (Open Problem 1) and
should be treated as such.


/**** Performance Optimizations ****/

This section specifies performance optimizations that are protocol-
compatible and expected of conforming implementations. They are not
optional for block-producing nodes: violating them means failing to
produce blocks within the block time.

1. Parallel batched fragment verification

Before beginning binary tree aggregation, the block producer holds N
per-fragment proofs. Each proof verification is independent of all
others. Block producers MUST verify all N fragment proofs in parallel
before beginning any aggregation step.

On GPU hardware (ICICLE backend), all N verifications SHOULD be
dispatched as a single batched kernel call. The GPU kernel schedules
N independent verification pipelines across available compute units.
At N = 500 and ~5 ms per verification single-threaded, naive serial
verification would take 2.5 seconds. Batched GPU verification at
500x parallelism reduces this to the latency of a single verification
plus scheduling overhead: approximately 10-30 ms total.

On CPU, implementations SHOULD use thread-pool parallelism with one
verification per thread. With 16 CPU cores: 500 verifications at
20 ms each = 625 ms wall-clock (acceptable as a pre-aggregation step
on a CPU-only fallback node, though GPU is required for target
throughput).

The proof verification step precedes the aggregation tree. No
aggregation level begins until all leaf proofs are verified.

2. IMT nullifier pre-sorting for batch insertion

Inserting N nullifiers into the IMT sequentially is O(N log N) in
path operations, with each insertion requiring a tree path traversal
to find the insertion point. The IMT's sorted structure allows a
more efficient batch strategy.

The block producer already possesses the full set of new_nullifiers[]
for the block before the state_diff is finalized. The block producer
MUST pre-sort new_nullifiers[] in ascending order before including
them in the BlockCandidate.

Benefit: when the node applies the sorted nullifier list to the IMT,
it can use a sequential scan strategy rather than individual binary
searches for each insertion. Consecutive insertions into nearby
positions in the sorted tree share subtree paths, enabling node
reuse across multiple insertions. This reduces the number of distinct
tree node reads for a batch of N sorted insertions from O(N log N)
individual reads to approximately O(N + log N) amortized reads for
a sorted batch, a significant practical speedup at N = 500-1000
per block.

The canonical ordering of new_nullifiers[] in the state_diff is
ascending numeric value. This is both the IMT insertion order and
the storage order. Block validators apply nullifiers in this order.

3. MMR batch append

Each block appends up to 1,000 new output commitments to the MMR.
Naive sequential appending updates the MMR peak structure 1,000
times with intermediate root recomputations that are discarded.

The MMR supports efficient batch appending: given a batch of M new
leaves, the final peak structure can be computed in O(M) hash
operations by processing all new leaves simultaneously.

Algorithm (batch MMR append for M new leaves):

  1. Represent the current MMR as its peak list: a set of perfect
     binary trees at heights h_0 > h_1 > ... > h_k where each
     height has exactly one peak.
  2. Append all M new leaves to a pending list.
  3. While the pending list contains two trees at the same height,
     merge them into a tree one height taller (one hash operation).
  4. Merge the pending list with the existing peak list using the
     same height-merge rule.
  5. Compute the new MMR root as the hash of all remaining peaks.

This requires exactly M - 1 + (merges into existing peaks) hash
operations, versus up to M * log(N) hash operations for sequential
appending. At M = 1,000 and N = 100,000,000: batch requires ~1,000
hashes; sequential requires ~27,000 hashes. A 27x reduction.

Implementations MUST use batch append for all block-level MMR
updates. Per-output sequential appending is not permitted in block-
producing or block-validating code.

4. Aggregation pipeline overlap

The binary tree aggregation has ceil(log2(K)) sequential levels, but
within each level all pairwise aggregations are independent. GPU
implementations SHOULD exploit this by:

  - Dispatching all pairwise aggregations at a given level as a
    single batched GPU kernel.
  - Beginning the next level's kernel dispatch as soon as the
    current level's outputs are available.
  - Overlapping proof I/O (loading proof blobs from memory) with
    GPU computation using CUDA streams or equivalent.

At K = 512 (9 levels), naive sequential pairwise aggregation would
run 511 aggregations in series. Level-parallel dispatch runs 256
aggregations in the first batch, 128 in the second, etc. — 9 GPU
dispatches total. If a single aggregation takes 200 ms on GPU, the
sequential approach takes ~102 seconds; the level-parallel approach
takes ~9 x 200 ms = ~1.8 seconds (idealized; actual performance
depends on parallelism within each level's batch).

5. Witness bundle prefetching optimization

Full nodes SHOULD prefetch and maintain updated WitnessBundles for
all outputs in their local mempool (outputs referenced by active
SpendIntents). When the chain advances by one block:

  - All MMR paths for existing outputs gain one position in the MMR.
  - Non-membership paths for uncreated nullifiers remain valid if
    no new nullifier from the new block falls between L.value and
    R.value in the IMT sorted order.

For the majority of outputs in the mempool, the non-membership
witness remains valid across block boundaries. Only outputs whose
IMT witnesses are invalidated by a new nullifier insertion require
a full witness refresh. Nodes SHOULD compute which IMT witnesses
are invalidated by each new block's nullifier set and send targeted
witness refresh bundles only to wallets whose witnesses were
affected, rather than broadcasting fresh witnesses for all outputs.

This reduces witness relay bandwidth from O(N_mempool) per block to
O(N_invalidated) per block, where N_invalidated << N_mempool in
normal operation.


/**** Future Work ****/

1. Circuit minimization and formal audit

A machine-checked Lean4 or Coq specification of the spend circuit AIR,
verified against the prose protocol semantics. This is not "future
work" in the aspirational sense — it is a prerequisite for mainnet.

2. Better proving backends

WHIR (Arnon, Chiesa, Fenzi, Yogev, EUROCRYPT 2025, ePrint 2024/1586)
achieves approximately 1.95x smaller proofs than FRI (157 KiB vs
306 KiB) and approximately 3.8x faster verification (1.0 ms vs 3.9 ms).
STIR (Arnon, Chiesa, Fenzi, Yogev, CRYPTO 2024 Best Paper,
ePrint 2024/390) achieves 1.25-2.46x over FRI. Neither is yet
integrated into a production recursive ZK stack as of this writing,
but WHIR in particular is a strong candidate for a future Specter
proof backend once production recursive composition is available.

Concrete impact of WHIR on Specter if adopted:

  Per-fragment proof size: ~30-50 KB (from 50-100 KB) — smaller
    fragments mean lower relay bandwidth (~60-70 KB/s sustained
    at full blocks, vs ~125-417 KB/s for FRI).

  Block verification latency: ~1-5 ms (from 5-20 ms) — 3.8x
    improvement benefits all nodes including light clients.

  Epoch certificate size: ~25-100 KB (from 50-200 KB).

  Aggregation time: WHIR's smaller proof sizes reduce the I/O cost
    of aggregation tree steps. Expected improvement in block
    aggregation: ~1.5-2x.

WHIR integration would require a protocol upgrade: the proof
format, public input encoding, and verification circuit would all
change. This must be planned as a clean hard fork, not an in-place
patch.

Circle STARKs over M31 (Stwo, deployed on Starknet mainnet on
November 4, 2025) achieve 940x higher Poseidon2 proving throughput
than the Stone prover on equivalent hardware. Stwo is a candidate
backend for future integration as its recursive composition tooling
matures.

Concrete impact of Stwo on Specter if adopted:

  Per-fragment CPU proving time: potentially sub-second for W1
    circuits (from 1-5 seconds). At sub-second CPU proving, the
    two-phase spend path's Phase 2 delay approaches the network
    round-trip time, making the UX nearly indistinguishable from
    standard signature-based transactions.

  Block producer hardware requirements: if per-fragment proving drops
    to sub-second on CPU, the GPU aggregation bottleneck shifts.
    A 120-second block time with sub-second leaf proofs could
    support substantially higher fragment counts per block — or
    relax GPU requirements for block production.

  Block throughput ceiling: with 0.1-second CPU proving per
    fragment, the bottleneck shifts from proof generation to proof
    relay bandwidth and state-diff application. Fragment counts
    of 5,000-10,000 per block may become plausible with a Stwo
    backend, contingent on aggregation and relay improvements.

Stwo integration requires the same clean hard fork consideration
as WHIR. The two backends should not be mixed within the same
epoch.

3. Conditional outputs

Timelocks, hashlocks, and threshold authorization can be expressed
as additional STARK circuit constraints rather than scripts. These
are v2+ scope.

4. Payment channels

Off-chain state channels with on-chain settlement fragments are
natural for this architecture and necessary for throughput beyond
the base-layer proving bound.

5. Epoch prover coordination refinement

The v1.3 epoch prover bounty provides basic economic incentives for
certificate production. Future versions may add a commitment scheme
(provers announce intent before proving, reducing duplicate work) or
a multi-prover reward split for partial proofs.

6. Dynamic fragment cap

Specter v1 fixes the block fragment cap at MAX_FRAGMENTS_PER_BLOCK =
512 (500 target, 512 as the tree ceiling). This cap is calibrated
against current GPU aggregation performance under Plonky3/BabyBear.

As hardware improves and proving backends improve, this cap will
become conservative. A fixed cap that cannot be raised without a
hard fork creates unnecessary protocol ossification.

A future version SHOULD introduce a mechanism for the fragment cap
to be adjusted via a network-signaled parameter upgrade, analogous
to Bitcoin's block size signaling but tied to measurable aggregation
performance rather than miner preference. The parameter is:

  MAX_FRAGMENTS_PER_BLOCK   (v1: 512)

The corresponding consensus tree ceiling is always the smallest
power of 2 >= MAX_FRAGMENTS_PER_BLOCK.

Upgrade condition: an increase in MAX_FRAGMENTS_PER_BLOCK is valid
only if a supermajority of recent block producers (measured over the
prior EPOCH_LENGTH blocks) demonstrate they are producing
EpochCertificates and block proofs within their respective time
windows at the new cap. This ties the cap upgrade to empirical
demonstrated capability, not theoretical hardware estimates.

/**** Summary ****/

What Specter removes

  - ECDSA, Schnorr, ML-DSA, SLH-DSA from the authorization layer
  - permanent protocol-level public-key authorization artifacts
  - trusted setup ceremonies
  - indefinite per-block proof storage

What Specter keeps

  - transparent balances and addresses
  - publicly auditable supply
  - permissionless block production (PoW)
  - trustless historical verifiability (via epoch proofs)

What Specter uses instead

  - hash-derived addresses (no algebraic public-key structure)
  - witness-based ZK spend authorization
  - nullifiers for permanent spentness memory
  - recursive STARK aggregation for blocks
  - epoch compression proofs for historical validity
  - anchor-window binding for practical mempool economics
  - two-phase spend path for UX
  - witness bundle protocol for censorship-resistant send preparation
  - two-role block production with consensus-enforced fee distribution

What Specter buys

  - no permanent public authorization material on-chain
  - no PQ signature-size bloat
  - O(1) block verification independent of fragment count
  - trustless historical bootstrapping via epoch proofs
  - quantum-surface reduction: no Shor-targetable keys in permanent state

What Specter pays

  - proving latency at send time (1-5 seconds CPU; mitigated by
    two-phase path and rolling witness cache)
  - permanent nullifier-state growth (~84 GB at year 10 at target
    throughput; substantially less than signature alternatives)
  - block-producer hardware asymmetry at the proving layer
  - epoch proof production requirement (~16 MB/year additional permanent
    storage; trades per-block proof storage for compact epoch certs)
  - complex wallet infrastructure vs. signature chains

That is the actual trade.


/**** Conclusion ****/

A quantum computer cannot apply Shor's algorithm to a public key
that does not exist.

That observation is the starting point of Specter.

Specter does not try to salvage the signature-chain model by
replacing one permanently-stored authorization artifact with a
larger one. It removes the structural pattern. Authorization
evidence exists during consensus and then is replaced by compact
epoch validity certificates, not discarded into an unverifiable void.

In Specter, ownership is demonstrated by proving knowledge of a
secret witness, not by revealing a public key and verifying a
signature. The chain retains state. Authorization artifacts are not
permanently on-chain. Historical authorization correctness is
provably verifiable through compact epoch certificates.

This comes at a price.

Proof generation costs 1-5 seconds. Nullifier state grows permanently.
Block production hardware is asymmetric. The epoch prover infrastructure
requires ongoing operation and explicit economic incentives. The circuit AIR must be formally audited
before deployment.

These are not defects in the prose. They are the actual engineering
costs of this architecture.

Specter is a claim of direction:

  If permanent public-key authorization artifacts are the wrong
  shape for the quantum era — too large, too permanent, too Shor-
  targetable — then the right move is not to resize them.
  The right move is to stop storing them, and prove correctness
  by another means.

That is what Specter does.
